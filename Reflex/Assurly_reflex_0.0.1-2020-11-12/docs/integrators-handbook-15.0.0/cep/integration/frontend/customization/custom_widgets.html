Creating custom question widgets

A widget is an HTML and JavaScript snippet (i.e. Angular directive) that maps a single question in JSON format to a graphical representation that is editable by the user. Example: The number widget is a directive that takes a question of type number and renders an input field.

Responsibilities of a custom “widget”:

  • mapping the user input to a value of the correct type and putting it in the application JSON
  • has to be switchable to read-only (not plain-text)
  • has to do the validation (except for required)
  • has to tell the system the moment the user has finished the input (the moment when a round-trip may be triggered and validations are shown)
  • must adapt values that are set from outside

On the other hand, a template is a wrapper that also shows the validations and renders the sub-questions. A template is a core part of the questionnaire and can’t be customized. Therefore complex constructs (i.e. structs and lists) are not customizable.

KB pre-requisites

There has to be a special configuration in the KB to display custom widgets. A question must be tagged with type “custom”, and a directive name has to be set. Optionally you can specify more key-value pairs that will be passed to your directive.

AssessmentVariable CustomWidget {
	type Decimal
	tagged RenderHint = "{ \"type\":\"custom\", \"directive\":\"custom-height\", \"key\":\"value\" }"
}

The directive value is the name of the HTML tag. During runtime, the directive will be called like this:

<my-custom-widget ...></my-custom-widget>

Development

You have to create an AngularJS directive inside your portal frontend. The custom directive has to have an isolated scope and the DOM-restriction “E”. You can use the following boilerplate:

angular
	.module('myPortalApp')
	.directive('myCustomWidget', MyCustomWidget);

function MyCustomWidget() {
	'use strict';

	return {
		restrict: 'E',
		scope: {
			widgetProperties: '<',
			widgetApi: '<'
		},
		templateUrl: 'templates/my-custom-widget.html',
		link: function (scope) {
			//your code here
		}
	};
}

If you don’t have an AngularJS portal app (Scenario 3), you can register the directive directly at the reflexCepQuestionnaire module. As you can see, there are two objects passed to the isolated scope of the directive.

Widget Properties

Key Description
question The question as plain JavaScript object. Do not modify this object. Please use setValue(value) to set an answer.
readOnly A boolean that indicates whether your custom input should be disabled. Since the questionnaire can be switched to readonly at any time, this value should be watched when used outside your template.
language The current questionnaire language key. If you use a translate service in code rather than in the template, remember to watch this value.
renderHints An object containing all the RenderHints from the question including key: value from the KB example above.

Widget API

Key Description
setValue(value) Sets the new answer of the question.
commit() Has to be called when the input is done, which may be after a button click or after an input field has been blurred. When this function is called, the question validation errors are shown. Reflexive questions may trigger a round-trip now.
onExternalChange(listener) There are scenarios where the answer of a question may be set from outside your directive. You can register a listener that is called with the new value to sync the UI.
addValidator(key, validator) Every time the answer is changed, all validators are called in the registration order. If there is at least one that returns a falsy value, the question is invalid and the questionnaire may not be submitted. The key parameter is the translation key which must not contain dots (.). Only the required-validation is done by the system.

Changing the question’s answer

To change the answer of the question, simply call setValue(newValue). Keep in mind that the type has to match the questions type (integer for type ‘number’, string for type ‘text’, etc.).

Gaining focus

When the user tries to submit an invalid questionnaire, the first invalid question is focused. This could be your directive! To tell the system which element should be focused in this case, just add the class focusable. Example:

my-custom-widget.html

<input id="inputPart1" />
<input id="inputPart2" class="focusable" />
<input id="inputPart3" class="focusable" />

The input with ID “inputPart2” will be focused, because it is the first one that matches the selector. There has to be at least one focusable element.

Reflexive questions

If the question you’re decorating is a reflexive question, there may be changes in the questionnaire when the application is synced. This synchronization has to be explicitly triggered by your directive by calling the commit function, because only you know when the user is done with the input.

Validation

CEP only provides the validation for required. If no answer is given by your widget and the question has the required constraint, your component is invalidated and an error message is shown. If you have server validations, hints or warnings, those will be shown as well as soon as the application has been synced.

To add custom validations, you can use the addValidator(key, validator) function. The function you register there takes the current value as input and has to return a truthy or falsy value. Everytime the answer changes, this function will be called. If it returns a falsy value, an error message will be shown. The text comes from the translation file under components.[questionType].errors.[yourKey]. Invalid answers are not saved to the model and won’t be sent to the RAS.

Read-only

Every question can switch anytime into read-only mode. Please $watch widgetProperties.readOnly or use constructs like

<input type="text" ng-disabled="widgetProperties.readOnly" />

Helpful tools and services

Translation

To provide custom translations for your widget, for example, when none of the provided traslations are suitable, you can extend your translation file with custom keys. In order to do this, you need to extend the component namespace of your translation with either a custom widget namespace or extending a pre-existing namespace. In the former case, your translation file could look like this:

components: {
	...
	
	'my-custom-widget': {
		'help-text': 'some useful information',
		...
	}
	
	...
}

Please note: validation error messages must always reside under the namespace of the question type you have provided a custom widget for. For example, a custom widget for question type ‘number’ always uses the translation namespace components.number.errors for validation error messages (see validation chapter above).

The translation mechanism provided by CEP is triggered by the cepTranslate filter, which can be used in your template file for your custom widget like so:

<span>{{ 'my.translation.key' | cepTranslate : widgetProperties.language }}</span>

The language parameter for the filter (that is, the expression after the :) can either be provided directly (for ex. cepTranslate : 'de-DE') or by using the provided language in the widgetProperties parameter.

Component blur observer

If you have a complex widget that has more than one input element (e.g. feet/inches widget), it may be hard to tell, when the user is done with the input because blur may be just an input field switch. Worry no more! CEP has already a built-in solution for that called: CEP Component Blur Observer. Just wrap your widget inside a container (<div>) and use the CEP Component Blur Observer like in this example:

my-custom-widget.html

	<div class="my-custom-widget" cep-component-blur-observer after-action="widgetApi.commit">
		<input id="inputPart1" />
		<input id="inputPart2" />
		<button type="button" blurs-component>Commit manually</button>
	</div>

With this code, the commit-function will be called as soon as the user clicks somewhere outside the container. If you want to blur the component from certain elements inside the container, just add the blurs-component attribute.

Example

In this example we want to create a widget with three configurable buttons that change the answer of a money question to a fixed value that comes from the RAS via a RenderHint.

applicationForm.kwl

...
AssessmentVariable CustomMoneyWidgetQuestion {
  type Integer
}
require CustomMoneyWidgetQuestion {
	tagged RenderHint = "{ \"type\":\"custom\", \"directive\":\"custom-money\", \"values\":\"5000,10000,20000\" }"
}
...

yourPortal/scripts/customMoney.directive.js

angular
	.module('MyPortalApp')
	.directive('customMoney', CustomMoneyWidget);

function CustomMoneyWidget() {
	'use strict';

	return {
		restrict: 'E',
		scope: {
			widgetApi: '<',
			widgetProperties: '<'
		},
		templateUrl: 'templates/custom-money.html',
		link: function (scope) {
			//This function will be called on button click.
			//Since we have isolated user actions, the value is directly committed.
			scope.changeValue = function(value) {
				scope.widgetApi.setValue(value); //change the question's answer
				scope.widgetApi.commit(); //trigger validation and loading of reflexive questions
			};
		}
	};
}

yourPortal/templates/customMoney.tpl.html

<div class="btn-group">
	<button type="button"
			class="btn btn-default focusable"
			ng-disabled="widgetProperties.readOnly"
			ng-repeat="value in widgetProperties.renderHints.values.split(',')"
			ng-click="changeValue(value)"
			ng-class="{active: widgetProperties.question.answer == value}">
			{{value}} $
	</button>
</div>˘

All requirements are met through code inside the template:

  • The buttons watch the widgetProperties.readOnly value and get disabled when true.
  • The buttons use the RenderHints from the KB to determine their values.
  • The buttons also watch the question’s answer to gain ‘active’-states.
  • All of the buttons are focusable so the first one gains focus when this is the first invalid question.
  • External changes are reflected by: widgetProperties.question.answer == value which means that the button corresponding to the question’s answer gets active.
