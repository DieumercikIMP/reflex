Extending questionnaire translations

Most of the translations visible to the user are generated by the underlying knowledge base such as question texts, validation messages, answer options, etc. However, the translations only applicable to the user interface of the questionnaire, e.g. the client error messages must be configured in the front end code, more specifically in the integrating AngularJS app. For this, the `reflex-cep-questionnaire` provides the module `cepTranslationModule` containing a `provider` that can be injected in the configuration phase of the app to extend the existing translations or apply new ones.

Step 1: Configuring a new language

There are two ways to register a new language in the questionnaire: Either directly by appending an additional `config` call to the main module or indirectly by providing a new module. The latter is the more elegant and therefore preferred way to do the registration.

First, a new javascript file containing the following setup skeleton must be created and included:

angular
	.module('seTranslation', ['reflexCepTranslate'])
	.config(function(cepTranslateProvider) {
		cepTranslateProvider.registerLanguage('se', translationTable);
	});

var translationTable = {
	components: { ... },
  	products: { ... }
}

The new module must declare a dependency to reflexCepTranslate that contains the cepTranslateProvider. This provider can register a new language through a method call to registerLanguage. Its parameters are a language code and a translation table, where the former must be of the format <language-code>-<country-code>-<variant> while the latter is a simple JSON containing the namespaces components and products. To activate the language in the integrating app, the module must be registered as dependency in the main module.

When not using AngularJS as the portal frontend-framework, the module definition slightly differs. Instead of defining a new module, the config can be appended to an already existing module, either reflexCepTranslate or reflexCepAngularApp. The latter would be the preferred option:

angular
	.module('reflexCepAngularApp')
	.config(function(cepTranslateProvider) {
		cepTranslateProvider.registerLanguage('se', translationTable);
	});

var translationTable = {
	components: { ... },
	products: { ... }
}

With the language code format given above it is possible to register a hierarchy of translations where the most specific variant language will inherit from the more general country language which in turn inherits from the most general language. As a fallback and base of every language hierarchy en is provided by the questionnaire and must not be registered. Additionally, the questionnaire ships with a module ‘defaultGermanTranslation’ which registers German default translations with the language code de unless there is already a translation registered with that code. Contrary to the language code en, it is both possible to re-register your own translation for code de or extend it.

Example

The following languages shall be registered in the questionnaire: de , fr , de-DE-berlin , es-MX-az. The resulting language hierarchy will then look like this: en ⟵ de ⟵ de-DE-berlin, en ⟵ fr, en ⟵ es-MX-az. Please note: de-DE-berlin would inherit from de-DE. But because de-DE is not registered, it will rather inherit from de, the more general language matching the given hierarchy defined in the language code. es-MX-az will inherit only from en, because none of the more general languages (es-MX or es) are registered.

The order of language registration can be arbitrary. The cepTranslateProvider will always find the correct hierarchy. Also, it is possible to construct a different hierarchy by providing a prototype language as the third parameter to the registerLanguage method call. The given language will then extend this prototype language instead of one defined in its language code hierarchy.

Extending a registered language

It is also possible to extend an already registered language. This can be necessary for example when `cepTranslate` is used for other components that have a different set of translations or to override translations in `en`. To achieve this, the `cepTranslateProvider` offers the method `extendLanguage(lang-code, translationTable)`. `lang-code` refers to the language code described above, `translationTable` is an object defining new translations.

Therefore it is possible for a variant language to override just some of the keys and use a more general language for the rest.

Step 3: Setting the questionnaire language

The questionnaire has two mechanisms to set the language: Either by the attribute `language` on the directive or by the URL query param `lang`. The former should be used, when the questionnaire should only support one language; the latter in all other cases.

Examples

When choosing the attribute on the questionnaire directive, the integration code should look like this (omitting other attributes for simplicity): 
<cep-questionnaire language="'de'"></cep-questionnaire>

When using the lang parameter, the page, on which the questionnaire is integrated, should be called with the lang parameter provided, for example /assessment?lang=fr. The integration code then has to look like this (again, omitting other attributes):

<cep-questionnaire language="cepLanguage"></cep-questionnaire>

cepLanguage will be a rootScope variable and is kept in sync with the URL parameter lang by the questionnaire. Therefore, whenever the language should be changed, the parameter lang of the URL must be modified.

Configuring date formats

Since the date format depends on the current language, they are translation keys instead of configuration keys although they are never visible in clear text to the user. The `format` key is the configuration for the date picker while the `placeholder` key is what is displayed to the user in an empty input field. If you want an empty placeholder, provide an empty string (`''`).

The date formats are Angular Bootstrap specific. (read more)

date: {
	...,
	formats: {
		year: 'yyyy',
		month: 'MMMM yyyy',
		day: 'dd/M/yyyy'
	},
	placeholder: {
		year: 'YYYY',
		month: 'eg. "January 1970"',
		day: 'DD/MM/YYYY'
	}
}

The uib dateparser has two special format strings:

  • ‘M!’ which means: ‘month as number with optional leading zero’, e.g. ‘4’ or ‘04’ for April and
  • ‘d!’ which means: ‘day as number with optional leading zero’, e.g. ‘1’ or ‘01’ for the first of a month

Translating error messages

The questionnaire will notify the surrounding AngularJS app by broadcasting the questionnaire:notification event on the rootscope. This event carries a payload with the following keys:

Key Description
type The type of notification. Either info, error or warning.
message Translation key for the message. See below for further details.
headline Translation key for the headline. See below for further details.
error An error object. Mostly a HTTP error object.

The message and headline keys of the event payload are not translated by the questionnaire but must be provided by the integrating AngularJS app. Currently only error notifications will be send out containing keys from the following JSON:

error: {
	saveApplication: {
		headline: 'Error',
		message: 'Could not handle application data.'
	},
	loadApplication: {
		...
	},
	loadProducts: {
		...
	},
	completeTask: {
		...
	},
	loadSearchResult: {
		...
	},
	notoken: {
		...
	}
}
