XSL FO templates

This section describes DCS specific configuration and features concerning XSL-FO templates. They can be applied by reusing the dcs:xslTransformation pipeline element.

Multi language support

Translation texts that can be used within the template to support multiple languages by processing the input with the dcs:i18NXmlProcessor pipeline element.

Structuring the stylesheet

XSL-FO stylesheets can be structured by moving parts to other files which can be included into the original file. A include would be configured in the following way:

<xsl:include href="includes/general_templates.xsl"/>

The path is relative to the main XSL FO stylesheet.

Images can be embedded by using the the XSL FO url function. The base url for look up is the template directory of the currently processed template. The following example will load the logo.svg image from the template root dir:

<fo:block>
  <fo:external-graphic src="url('logo.svg')"
    width="150px" content-width="scale-to-fit"
    content-height="scale-to-fit" />
</fo:block>

Adding custom fonts

Custom fonts / font families that should be used during the PDF generation can be configured within the template files, e.g. inside the i18n.xml to be language dependent.

<i18n language="en">
  <text id="fo.font-family">AjarSans-Bold</text>
  <text id="fo.script">auto</text>
  ...
</i18n>

To use font families that are not supported by FOP you have to provide those font files as well as a FOP configuration file which tells the fop-factory where to find those fonts. The expected name of this config file is fopConfig.xml. The DCS is looking for it inside the directory of the specific document type (e.g. <reflex.config.dir>/<dcs.config.dir>/doctypes/sample-application). Down below you can see an exemplified file:

<fop version="2.2">
  <!--  Strict user configuration  -->
  <strict-configuration>true</strict-configuration>
  <!--  Strict FO validation  -->
  <strict-validation>false</strict-validation>
  <!--  Base URL for resolving relative URLs  -->
  <base>./</base>
  <renderers>
    <renderer mime="application/pdf">
      <fonts>
        <!--  register fonts  -->
        <directory>fonts</directory>
        <!-- automatically detect operating system installed fonts -->
        <auto-detect/>
      </fonts>
    </renderer>
  </renderers>
</fop>

Due to some limitations and bugs of the current Apache FOP version the configuration for external fonts is functional but limited.

There are two ways to successfully define the <directory> for a renderer.

  • Relative to the working directory of the Java process running dcs. This may be varying on your tomcat installation.
  • An absolute path of the fonts directory.

Important: In contrast to prior documentation neither <base> nor <font-base> influence the font resolution.

Inside the next example the font lookup path is set to an absolute path pointing to the common/fonts folder inside the <dcs.config.dir>.

<fop version="2.2">
  <!--  Strict user configuration  -->
  <strict-configuration>true</strict-configuration>
  <!--  Strict FO validation  -->
  <strict-validation>false</strict-validation>
  <!--  Base URL for resolving relative URLs  -->
  <base>./</base>
  <renderers>
    <renderer mime="application/pdf">
      <fonts>
        <!--  register fonts  -->
        <directory>/opt/apache-tomcat/conf/dcs/common/fonts</directory>
        <!-- automatically detect operating system installed fonts -->
        <auto-detect/>
      </fonts>
    </renderer>
  </renderers>
</fop>
