AppQueue

Hannover Re provides AppQueue as a software component for cedents to upload application data as JSON documents to the ReFlex ingestion endpoint. AppQueue runs as an ongoing process until it receives an interrupt signal from the OS. It is supposed to run as a background task (or daemonized process). While running, it does the following:

  1. Observes a given directory, watching for new application data,
  2. Sends the files to the server endpoint via HTTPS POST (with retry mechanics in case of an error).
  3. Moves the completed files to a given target directory, or into a special directory if an error occured.

Prerequisites

Recommended Systems and required Java Version

An overview of the recommended systems and the required Java version to run AppQueue can be found here: ReFlex recommended systems

Memory and CPU

We recommend to use at least 512MB memory:

  • java -Xms256m -Xmx512m
  • The AppQueue will need up to 25% of a CPU if busy.

Connectivity

The server endpoints must be reachable from the cedent’s network
  • PREP (preproduction) https://reflex-mi-prep.hannover-re.com/upload/(port:443)
  • PROD (production) https://reflex-mi.hannover-re.com/upload/(port:443)

Using AppQueue for file upload

AppQueue - secure upload client

Overview

Logging: In case application data cannot be sent (after a configurable number of delayed retries), problems will be logged and the corresponding file will moved to a special problematic directory for later analysis. The problematic directory name can be configured.

Input: AppQueue can deal with flattened or recursive input directories and filter the files to upload based on regular expressions. AppQueue works as Unix-style daemon process, i.e. it starts but does not return. It waits for files to upload and process until terminated manually.

AppQueue and its source code can be provided upon request, so the code can be used as a basis for integrating its anonymisation and upload processes into other environments.

Documentation: Development documentation is available upon request from Hannover Re.

Processing error: If an error occurs while processing application data the problem will be logged and the corresponding file is moved to the problematic folder.

Please note: The problematic directory will not be emptied automatically. You have to empty the directory regularly.

Configuration

The following default values are used by AppQueue:

# ~~ The URL to which the application get POSTed (for prep: reflex-mi-prep)
ingest.endpoint=https://reflex-mi.hannover-re.com/upload

# ~~ Retry policies
ingest.maximumRedeliveries=3
ingest.redeliveryDelay=5000

# ~~ One-time execution mode

# determines whether AppQueue will be executed until all routes are processed one time or AppQueue constantly will run in the demon mode
ingest.onetimeExecution.mode=false
# set the time out until AppQueue will wait for a route to finish until the route will be finally shut down
ingest.onetimeExecution.shutdownStrategyTimeOut=300

# ~~ Inbox
# Directory listening for incoming applications
ingest.directory.in=/tmp/app-queue/inbox

# false: moves into sub-directory ".camel of inbox, true: deletes the file in the inbox directory
ingest.directory.in.deleteFileAfterProcessing=true

# Wether to look also in sub-directories relative to ingest.directory.in
ingest.directory.in.recursive=true

# Only care about XML files
ingest.directory.in.includesPattern=**/*.xml

# Processed Files (anonymized/problematic) will stick in sub-directories as given (or flattened)
ingest.flattenSubDirectories=false

# To which directory should applications de moved in case a processing error occured
ingest.directory.problematic=/tmp/app-queue/problematic

# Point to the SSL configuration (see "Security/Certificate Handling")
ingest.endpoint.sslConfig=ssl-configuration.xml

Any of those options can be overridden by providing an alternative property file like application-dev.properties upon startup. See Operating/Commands section for more information.

AppQueue can be invoked

  • periodically (e.g. though cron daemon) or manually
  • run as a directory-listening daemon

General (global) values valid for all environments are stored in application.properties configuration file. By default, the local profile is used.

AppQueue will look for its configuration file in the directory specified by the system property (or environment variable  $REFLEX_CONFIG_ROOTDIR that has to be present.

Security/Certificate Handling

To be able to connect to the Hannover Re Prepoduction or Production environments, AppQueue must be configured to present a client-side certificate that has been signed by Hannover Re for authentication purposes.

To make use of the client SSL certificate in AppQueue you need to create a Java key store. See  KeyStores and TrustStores (docs.oracle.com) for a details on key stores and trust stores.

  • A Key Store is used to provide credentials to a server (i.e. to Hannover Re) for client authentication.
  • A Trust Store is needed for the client to verify credentials of the server (i.e. client confirms it’s really Hannover Re) before connecting.

First, you‘ll need to generate a PKCS12 key store for both the private key (provided by Hannover Re) and the signed certificate received from Hannover Re:

foo@bar:~$ cat Preproduction.crt Preproduction.key > Preproduction.pem
foo@bar:~$ openssl pkcs12 -export -in Preproduction.pem -out Preproduction.p12

You will prompted interactively for a password. Enter a password and keep track of it. You will need it later when configuring AppQueue’s connection. The most important security measure is limiting access to the private keys, certificates and keystore files.

Hannover Re’s server certificate, custom trust stores

The initial SSL protocol handshake between any https client request and the Hannover Re web server includes a server-side certificate announcing this service as a trusted Hannover Re service.

The standard Hannover Re https web server certificate (validating “CN=*.HANNOVER-RE.COM”) is issued by root Certificate Authority “CN=DigiCert SHA2 Secure Server CA”. On certain operating systems, this root certificate is not in the default list of trusted root certificates, so the client cannot validate the web server‘s certificate. As a consequence, connection requests fail due to an insecure communication channel.

If the OS running AppQueue does not include this CA by default, it must be added manually using a custom trust store.

Adding the missing root certificate to AppQueue:

If it is not present, please contact Hannover Re to provide you with the public certificate file. In the following steps,  <DigiCertCA>.pem refers to the root certificate which can be found here: https://www.digicert.com/digicert-root-certificates.htm Look for the “DigiCert SHA2 Secure Server CA” certificate and click “Download PEM”

The customer is responsible to keep the root certificate up to date, as soon as the root’s expiration date has been reached.

Issue the following command:

foo@bar:~$ keytool -import -file <DigiCertCA>.pem -alias <YourDigiCertAlias> -keystore <YourTrustStore>.jks

You will be prompted for a password interactively. Similar to the case with the keystore generation, keep track of this value. Password complexity is not crucial, especially in comparison to limiting access to the truststore file.

<camel:sslContextParameters id="appQueueSslContext"
       xmlns:camel="http://camel.apache.org/schema/spring"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://camel.apache.org/schema/spring http://camel.apache.org/schema/spring/camel-spring.xsd">
    <camel:clientParameters>
        <camel:secureSocketProtocols>
            <camel:secureSocketProtocol>TLSv1.2</camel:secureSocketProtocol>
        </camel:secureSocketProtocols>
        <camel:cipherSuites>
            <camel:cipherSuite>TLS_ECDHE_RSA_WITH_AES_128_CBC_SHA256</camel:cipherSuite>
        </camel:cipherSuites>
    </camel:clientParameters>
    <camel:keyManagers keyPassword="{keystore-pw}">
        <camel:keyStore type="pkcs12" resource="{keystore-filename}" password="{keystore-pw}"/>
    </camel:keyManagers>
    <camel:trustManagers>
        <camel:keyStore type="jks" resource="{truststore-full-path}" password="{truststore-pw}"/>
    </camel:trustManagers>
    <!-- /optional -->
</camel:sslContextParameters>

Edit the file to include the following variables:

  • {keystore-filename}.jks the name of the keystore file, e.g. keystore.jks.
  • {keystore-pw} the password entered when generating the keystore (note 2 locations in file)
  • (optional) {truststore-full-path} the full resolved path of the truststore file, e.g. /home/appqueue/reflex/mi-appqueue/truststore.jks
  • (optional) {truststore-pw} the password entered when generating the truststore.

Make sure that application[-{profile}].properties being used includes the following line telling appqueue how to locate the ssl-configuration.xml file:

ingest.endpoint.sslConfig=ssl-configuration.xml

Operating/Commands

To start AppQueue, execute the command below. It will start listening to applications in the configured incoming directory for processing and upload to the Hannover Re environment:

foo@bar:~$ java -Dfile.encoding=UTF8 -DREFLEX_CONFIG_ROOTDIR=[cfg-dir] -jar AppQueue-{version}.jar

Optionally, you can specify the following arguments:

  • --version prints out AppQueue version number
  • --dryrun tests whether the Camel context can be started successfully.

If you want to use a specific configuration profile, the configuration file name needs to match the pattern application-{profile-name}.properties.  For example, to activate a profile prod configured in file application-prod.properties use the following command:

foo@bar:~$ java -Dfile.encoding=UTF8 -DREFLEX_CONFIG_ROOTDIR=[cfg-dir] -jar \
target/AppQueue-{version}.jar --spring.profiles.active=prod

This will look for application-prod.properties in the $REFLEX_CONFIG_ROOTDIR for its configuration.

One-time execution of AppQueue

AppQueue is designed to run as a daemon, in case this behaviour not wanted, AppQueue can be started in one-time execution mode.

To start AppQueue in one-time execution mode you can set a property (see the section Configuration above) or start AppQueue with an additional argument from the command line: –onetimeExecutionMode

The command line argument will over rule the property file, if the property is set false.

Please note:

AppQueue currently does not support NTLM-based proxy authentications due a restriction in the http networking library. A future release of AppQueue may add this functionality, given customer relevance and importance.

Security / Proxy Configuration via URL Parameters

Proxy options can be defined as URL parameters in the ingest.endpoint property in the properties file, e.g.:

ingest.endpoint=https://reflex-mi.hannover-re.com/upload?proxyAuthUsername=<user>&proxyAuthPassword=<passwd>&proxyAuthMethod=basic&proxyAuthHost=<proxy host>&proxyAuthPort=<proxy port>&proxyAuthScheme=http

The following parameters are available:

  • proxyAuthHost - the proxy host name (mandatory)
  • proxyAuthPort - the proxy port (mandatory)
  • proxyAuthUser - the username to use for authentication (optional)
  • proxyAuthPassword - the password to use for authentication (optional)
  • proxyAuthMethod - the authentication method to use, either ‘basic’ or ‘digest’ (optional)
  • proxyAuthScheme - either ‘http’ or ‘https’ (optional)
Proxy Configuration via System Properties on the command line

You may also set the parameters listed above using the standard Java mechanism for setting system properties on the command line with the -D prefix. However, on the command line the parameters must be prefixed with https., e.g. https.proxyAuthHost

Furthermore the standard Java networking parameters for proxy configuration are supported on the command line: * https.proxyHost - the proxy host name * https.proxyPort - the proxy port

Take care to provide the Java executable parameters in the proper order, for example:

java -Dhttps.proxyHost=... -Dhttps.proxyPort=... -jar appqueue.jar

For further information please refer to the Java documentation for your platform and version, for example: https://docs.oracle.com/javase/8/docs/technotes/tools/unix/java.html
