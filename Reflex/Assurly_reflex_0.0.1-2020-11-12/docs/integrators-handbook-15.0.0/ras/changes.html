Release and migration notes

Summary

  • Exception when using keywords AND, OR, or NOT during assessment factor search
  • The logging framework has been changed from log4j to logback
  • Support for Logstash/GELF

Exception when using keywords AND, OR, or NOT during assessment factor search

  • Type: bug fix

When using one of the keywords AND, OR, or NOT (in upper-case letters) during an assessment factor search, the RAS would previously throw an exception that resulted in a 500 internal server error. This problem could occur in one of the following situations:

  • During a risk assessment, e.g. when evaluating the answer for an assessment factor search question.
  • During an assessment factor search, e.g. when an applicant is searching for an assessment factor representing a medical condition.

The problem has now been fixed. The RAS (or rather Apache Lucene) no longer interprets the words AND, OR, and NOT as keywords but rather as normal elements of the search term.

Please note that in any of these cases, the RAS implements its own transformations to convert a given search term into a valid Lucene query. Hence, integrators should not perform any transformations of the search term on the client side. The search itself is case-insensitive and can handle different spellings of a word depending on the locale (e.g. regarding the letters “e” and “è” in French).

The logging framework has been changed from log4j to logback

  • Type: change

The logging framework for RAS-REST has been changed from log4j to logback. Due to this change, the logging configuration must now be specified in a logback-spring.xml instead of a log4j.xml file.

Migration notes

The log4j.xml that has previously been used to configure the logging for RAS-REST must be replaced by a logback-spring.xml. Please refer to the documentation of the logging configuration for more information as well as a sample logback-spring.xml.

Support for Logstash/GELF

  • Type: change

The RAS-REST web application now supports Logstash/GELF appenders for sending log messages to a Graylog server. Please refer to our logging documentation
for more information.
