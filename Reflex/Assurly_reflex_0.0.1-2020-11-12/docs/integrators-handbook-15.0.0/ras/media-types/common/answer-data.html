Answer data

A simplified data format for specifying an answer to a question of a risk assessment data’s questionnaire. The answer data will be processed in certain processing steps involving risk assessments, e.g. processing of a provided questionnaire when performing a risk assessment.

This simplified answer format differs (at large) from the more complex answer structure used for the questions in a risk assessment data’s questionnaire. This is done in order to make the integration of external data as easy as possible for data-providing (third-party) clients.

Examples

General structure (simplified JSON)

{
    "meta": {...},
    "answer": ...,
    "overwriteMode": "overwrite-none"
    
    ...
}

Answer data with custom meta data (JSON)

{
    "meta": {
        "ui": {
            "hidden": true
        }
    },
    "answer": "John Doe"
}

Answer data with processing error (JSON)

{
    "meta": {
        "ras": {
            "processing": {
                "error": "Question could not be answer: Invalid answer format",
            }
        }
    },
    "answer": "John Doe"
}

Properties

meta (ObjectMetaData)

The optional meta data specified for the answer data.

The meta data specified by clients will be transferred to the meta data section of a question in the risk assessment’s questionnaire as soon as the answer data has been successfully processed. Thereby, the “ras” namespace will be ignored (i.e. not transferred). Further, the meta data for other namespaces will only be transferred if no corresponding namespace already exists in the question’s meta data section.

Meta data in the “ras” namespace may contain processing information, disclosing warning and error conditions that occurred when processing the answer.

answer (any)

The value with which a question should be answered. The actual format of the answer value depends upon the type of question to be answered. See descriptions of the supported questions types documented in the sections below:

  • Text question answer
  • Number question answer
  • Date question answer
  • Single-selection question answer
  • Multi-selection question answer
  • Assessment factor search question answer
  • Structure question answer
  • List question answer

This is a mandatory property that must be specified by clients. Answer values that to do comply to the expected answer format may be reported as errors in a processing info element attached to the answer data.

overwriteMode (String)

The overwrite mode specifies under which conditions an answer already given to a question will be overwritten by the answer data when it is processed. The overwrite mode may have one of the following values:

  • "overwrite-none": Never overwrite any answer already given to a question.
  • "overwrite-own": Overwrite an answer given to a question only if that answer has been given from the same source as the source providing the answer data.
  • "overwrite-all": Overwrite an answer given to a question in any case (i.e. independent of the source from which the question has been answered). Note: This mode should be used with care, since it prevents that answers of questions contained in a questionnaire effectively can be changed - to be more precise: answers may be changed but they will be “re-set” automatically to the provided answer every time a risk assessment is performed.

This is an optional property, that may be specified by clients. If no overwrite mode is specified by a client “overwrite-none” will be used as default.

Note: Depending on the type of question to be answered, additional properties might be required.

Text question answer

Example (JSON)

{
    "answer": "John Doe"
}

Properties

answer (String)

The answer value must be a valid JSON string.

Number question answer

Example (JSON)

{
    "answer": 85.5,
    "unit": "[kg]"
}

Properties

answer (Number or String)

The answer value must be a valid number. It must not have more than 101 integer digits and not more than 100 fractional digits.

Number values not matching the precision defined in the knowledge base will be accepted. Although rounding will occur in ReFlex/RAS internally when performing a risk assessment, this will not be reflected in the risk assessment’s questionnaire (i.e. the provided numerical answer will be transferred to the corresponding question in the questionnaire “as-is”). Clients can answer number questions by providing JSON numbers or JSON strings that can be converted into numbers.

unit (String)

This optional property can be provided to set the unit of the corresponding number question.

If the number question does support units, then the provided unit must be valid and available for this question, otherwise the answer is considered being erroneous and an error message is attached to it. If the number question supports units but no unit is provided, then the default unit is used.

Date question answer

Example (JSON)

{
    "answer": "1972-04-25"
}

Properties

answer (String)

The answer must be a string specifying a date in ISO 8601 format, according to the following patterns (depending upon the precision of the date as defined in the knowledge base):

  • year: yyyy (example: 2015)
  • month: yyyy '-' MM (example: 2015-01)
  • day: yyyy '-' MM '-' dd (example: 2015-01-30)
  • hour: yyyy '-' MM '-' dd 'T' HH (('+' | '-') HH ':' mm | 'Z')? (examples: 2015-01-30T12, 2015-01-30T12+01:00)
  • minute: yyyy '-' MM '-' dd 'T' HH ':' mm (('+' | '-') HH ':' mm | 'Z')? (examples: 2015-01-30T12:05, 2015-01-30T12:05+01:00)

Thereby,

  • yyyy is the year of the date (4 digits).
  • MM is the numerical month within a year (2 digits, ranging from 01 to 12).
  • dd is the numerical day within the month (2 digits, ranging from 01 to 31).
  • HH is the hour within a day (2 digits, ranging from 00 to 23).
  • mm is the minute within an hour (2 digits, ranging from 00 to 59).

For date values not matching the precision defined in the knowledge base an attempt will be made to convert the date to the correct precision (see below). If this is not possible the answer is considered being erroneous.

Note that in case no explicit time zone offset is specified, the date will be interpreted by RAS-REST as being in UTC time zone (see also general notes about time zones).

Date precision conversion

It is possible to answer a date question by providing a date with higher precision than required. If the given answer is compatible with the required precision, then the answer’s precision is reduced. If for instance a client provides an answer for a month question with the string “2015-11-26T01:39”, then the answer is converted into “2015-11”.

The following tables shows the supported precision reductions. For example, an answer provided with month precision may only be used to answer questions with precision year or month.

provided answer year question month question day question hour question minute question
year OK X X X X
month OK OK X X X
day OK OK OK X X
hour OK OK OK OK X
minute OK OK OK OK OK

Single-selection question answer

Example (JSON)

{
    "answer": "Male"
}

Properties

answer (String)

The answer must be a string specifying the ID of one of the options defined for the single-selection question that is to be selected.

An answer not matching an option’s ID is considered being erroneous.

Multi-selection question answer

Example (JSON)

{
    "answer": ["Cigarettes", "Pipes"]
}

Properties

answer (ArrayString)

The answer must be an array of strings with each element specifying the ID of one of the options defined for the multi-selection question that is to be selected.

An answer containing a string not matching an option’s ID is considered being erroneous.

Assessment factor search question answer

Examples

Answer data for assessment factors specified as IDs (JSON)

{
    "answer": ["HeartAttack", "Diabetes"]
}

Answer data for assessment factors specified as labels (JSON)

{
    "answer": ["heart attack", "diabetes"],
    "answerDescriptor": "Label"
}

Properties

answer (ArrayString)

The answer must be an array of strings with each element specifying either the ID or the label of an assessment factor that is to be added.

An answer containing a string for which no assessment factor can be identified is not necessarily considered being erroneous. Only if for none of the strings a corresponding assessment factor can be identified, the answer is considered being erroneous (i.e. if at least one assessment factor can be identified the answer is not erroneous).

answerDescriptor (String)

An additional description specifying how the answer value is to be interpreted. The answer descriptor is a string specifying whether the supplied answer value contains IDs or labels of assessment factors. Possible values are:

  • "Id": Each value given as an answer specifies the ID of an assessment factor.
  • "Label": Each value given as an answer specifies the label of an assessment factor.

This is a optional property, that may be specified by clients. If no answer descriptor is specified, the supplied answer values are interpreted as being IDs of assessment factors.

Structure question answer

Examples

Answer data without descriptor (JSON)

{
    "answer": {
        "Street": {
            "StreetName": "Rotherstr.",
            "StreetNumber": "19"
        },
        "Location": {
            "ZipCode": 15230,
            "City": ["FrankfurtOder"]
        },
        "Country": ["Germany"]
    }
}

Answer data with descriptor (JSON)

{
    "answer": {
        "Street": {
            "StreetName": "Rotherstr.",
            "StreetNumber": "19"
        },
        "Location": {
            "ZipCode": 15230,
            "City": ["Frankfurt an der Oder"]
        },
        "Country": ["germany"]
    },
    "answerDescriptor": {
        "Location": {
            "City": "Label"
        },
        "Country:" "Label"
    } 
}

Properties

answer (Object)

The answer must be an object with each property name specifying the ID of a structure’s member. Thereby, the ID may be scoped (e.g. “InsuredPerson.Street”) or un-scoped (e.g. “Street”). In case a scoped ID is given, the scope must match the scope of the structure variables that are to be answered. Each property’s value must be a valid answer value suitable for the structure’s member type. For example, if a member is a question of type text, then the property value must be a valid JSON string - if it is a question of type number, then it must be a valid number - and so on.

An answer containing a property name not matching the (possibly) scoped ID of any member of a structure is not necessarily considered being erroneous. The same holds true for a property’s value. Only if none of the structure members can be successfully answered, the answer is considered being erroneous.

It is also valid to provide answers only for a subset of a structure’s members. Please note that no “merging” of partially answered structure questions will be performed. If a structure question in a questionnaire is already (partially) answered, the provided answer for that question will be completely ignored - as long as no overwrite mode has been specified that forces the answer to be overwritten.

answerDescriptor (Object)

An additional description specifying how the answer value is to be interpreted. This is a optional property, that may be specified by clients.

The answer descriptor is an object with each property name specifying the ID of a structure’s member. Thereby, the ID may be scoped or un-scoped. In any case, the ID must be the same as the ID specified for the structure member’s answer value. Each property’s value must be a valid answer descriptor suitable for the structure’s member type.

List question answer

Examples

Answer data without descriptor (JSON)

{
    "answer": [
        {"ReadingSystolic": 120, "ReadingDiastolic": 80, "ReadingDate": "2015-01-01"},
        {"ReadingSystolic": 130, "ReadingDiastolic": 85, "ReadingDate": "2015-01-02"}
    ]
}

Answer data with descriptor (JSON)

{
    "answer": [
        {"Disease": ["diabetes"], "Drugs": ["insulin", "metformin"]},
        {"Disease": ["asthma"], "Drugs": ["bronchoretard"]}
    ],
    "answerDescriptor": {
        "Disease": "Label",
        "Drugs": "Label"
    }
}

Properties

answer (Arrayany)

The answer must be an array with each element being a valid answer value suitable for the list’s prototype question. For example, having a list of type “String” defined in a knowledge base (resulting in a prototype of text question), then the answer value must be an array of strings - if it is a number question, then it must be an array of numbers - and so on.

An answer containing an element not being an answer value suitable for the list prototype is not necessarily considered being erroneous. Only if none of the list’s elements can be successfully answered, the answer is considered being erroneous.

answerDescriptor (any)

An additional description specifying how the answer value is to be interpreted. This is a optional property, that may be specified by clients.

The answer descriptor must be a valid answer descriptor suitable for the list’s prototype question.
