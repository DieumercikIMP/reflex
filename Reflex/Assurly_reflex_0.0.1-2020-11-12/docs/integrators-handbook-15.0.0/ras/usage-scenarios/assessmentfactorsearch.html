Search assessment factors

The resource assessment factor search allows users of the RAS-REST-API to search for assessment factors. The found assessment factors can be added to risk assessments.

The user enters a search string and RAS-REST-API returns a list of matching assessment factors. The translated assessment factor labels and translated synonym labels are used for the search.

Scenarios:

  • Search diabetes with category
  • Search diabetes with question ID
  • Search with both question ID and category
  • Context sensitive search
  • Traverse all assessment factors of a category
  • Search with structures and list of structures

Finally, the last section highlights some internal details regarding the behaviour of the assessment factor search.

Scenario: Search diabetes with category

Search for Diabetes in the category Medical:

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=Diabetes&category=Medical

Scenario: Search diabetes with question ID

Search for Diabetes to answer a specific question with the ID PreviousDiseases:

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=Diabetes&questionId=PreviousDiseases

Scenario: Search diabetes using both question ID and category

Similar to the scenarios where a search is performed by either question ID or category, the search allows to specify both elements at the same time. This can be used to further restrict the search space, but may also lead to the search returning no results.

For the following example, consider an assessment variable named PreviousDiseases matching the category Medical, as shown in the following knowledge base definition:

AssessmentVariable PreviousDiseases {
    type AssessmentFactorSearch
    matches category==Medical
}

The following query will match both the question ID, and the category. The behaviour of this query is similar to the queries described in the scenarios “Search diabetes with category” and “Search diabetes with question ID”. Here the category specified in the query aligns with the matches clause in the definition of the assessment variable.

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=Diabetes&questionId=PreviousDiseases&category=Medical

The next example gives a query with the question ID again set to PreviousDiseases, but referencing the category Occupational. In this case, the matches clause of the specified search variable contradicts the specified category in the query, such that no assessment factors will be returned.

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=Diabetes&questionId=PreviousDiseases&category=Occupational

Scenario: Context sensitive search

In order to perform a context sensitive search, a question identified by its questionId as well as a questionnaire containing answers that are required for the search have to be specified in the request. For example, if question DiseasesWithContextGender only searches for medical conditions that match the tag GenderOnly==InsuredPerson.Gender, the questionnaire shown below containing an answer for question InsuredPerson.Gender could be specified:

{
    "questions": [
        {
            "type": "singleselection",
            "id": "InsuredPerson.Gender",
            "scope": "ASSESSMENTFACTOR:InsuredPerson",
            "meta": {
                "ras": {
                    "questionMapping": {
                        "sourceAssessmentVariableType": "SingleSelectionVariable"
                    }
                }
            },
            "options": [
                {
                    "id": "Male"
                },
                {
                    "id": "Female"
                }
            ],
            "answer":{ 
                "id": "Female"
            }
        }
    ]
}

Then, the following GET request to the assessment factors resource is answered with a HTTP status code 200 and a body containing all medical assessment factors matching Abortus and tag GenderOnly=Female:

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=Abortus&questionId=DiseasesWithContextGender&questionnaire={"questions":[...]}

Scenario: Traverse all assessment factors of a category

In order to get all assessment factors defined for a specific category, the “*” wildcard character may be used as a search term. For example, the following GET request to the assessment factors resource searches for all assessment factors defined for the Medical category.

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=*&category=Medical

In this case 2000 assessment factors were found, distributed over 20 pages. The result to the GET request contains the first 100 assessment factors, as shown below:

{
    "results": [ /*... first 100 assessment factors */ ],
    "page": {
        "number": 1,
        "size": 100,
        "totalPages": 20,
        "totalResults": 2000
    },
    "links": [
        {
            "href": "/rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors",
            "rel": ["next"],
            "method": "GET",
            "params": {"category": "Medical", "q": "*", "language": "en", "page": "2", "size": "100"}
        },
        ...
    ]
}

In order to get the remaining assessment factors, subsequent GET requests have to be send, using the provided “next” link. This has to be done until no “next” link is provided. Thus, the request for getting the next 100 assessment factors would be:

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=*&category=Medical&page=2&size=100

Scenario: Search for assessment factors where the question is part of a structure or the structure is part of a list

This scenario is similar to answering a specific question with a given ID. The difference is that if a assessment factor search question is part of a structure, the questionId is composed of multiple elements. The first element is the ID of the structure, StructureId in this example. The second element is the ID of the specific question, here with the ID PreviousDiseases. A search for the assessment factor Diabetes to answer a question PreviousDiseases that is part of a structure would look similar to the following example:

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=Diabetes&questionId=StructureId.PreviousDiseases

If the structure is used as the type of a List, the questionId is composed of three parts. The first element is the ID of the List, in this example ListId. The middle part is the fixed string prototype. The last part is the ID of the question, here with the ID PreviousDiseases.

The search for assessment factor Diabetes to answer the question PreviousDiseases that is part of a structure in a list would look like the following example:

GET /rs/knowledgebases/myknowledgebase/versions/1-0-0/assessmentfactors?language=en&q=Diabetes&questionId=ListId.prototype.PreviousDiseases

Internal behaviour of an assessment factor search

The RAS internally uses Apache Lucene in order to perform assessment factor searches. Thus, integrators should consider the following behaviour related to this.

Lucene query syntax

Lucene supports a range of term modifiers and boolean operators in order to adjust the search. Currently, the RAS is responsible for taking a search term provided for an assessment factor search and transforming it into a valid Lucene query. Any additional term modifiers or boolean operators inserted into the search term by the client are ignored during the search.

The only exception to this rule are the two wildcard characters ? and *. While the wildcard character ? matches a single character (e.g. the words text and test are both matched by the query te?t), the wildcard character * matches an arbitrary number of characters (e.g. the words risk, rook, and roadblock are all matched by the query r*k). By default, the RAS always appends the wildcard character * at the end of a search term.

Matching of search terms

Lucene has been configured to be relatively lenient when matching search terms with the translations for assessment factors. In particular, the matching is case-insensitive and can handle different spellings of a word depending on the locale (e.g. regarding the letters “e” and “è” in French).

For example, an assessment factor representing the medical condition diabetes may have the following translation in French:

Diabetes/fr = diabète

Then, any of the following search terms matches this translation:

diabète
DIABÈTE
diabete
Diabete
