Change product configuration

Scenarios:

  • Exchange product
  • Add rider
  • Remove rider

Scenario: Exchange product

Given assessed risk assessment data for a “CIC” product as shown below.

{
    "language": "en",
    "valid": true,
    "products": [{"id": "CICProduct", ... }],
    "questionnaire": {
        "questions": [
            {
                "id": "ApplicationDate",
                "scope": "GLOBAL",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": "2015-01-01",
                ...
            },
            {
                "id": "InsuredPerson.DateOfBirth",
                "scope": "ASSESSMENTFACTOR:InsuredPerson",
                "answerModification": { "source": "anonymous:Applicant", "timestamp": "2015-01-01T00:00:00" },
                "answer": "1975-10-02",
                ...
            },
            {
                "id": "CICProduct.InsuranceSum",
                "scope": "INSURANCEPRODUCT:CICProduct",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": 100000,
                ...
            }
        ]
    },
    ...
}

In order to exchange the “CIC” product with a “Life” product a POST to the product change resource is performed as shown below.

Thereby, the riskAssessment property of the request body’s JSON document contains previously assessed risk assessment data shown above.

The products property contains the configuration for the “Life” product to be changed to. Thereby, an insurance sum of 200 000 is specified for the “Life” product by providing a corresponding ID/value pair in the optional attributes object. Note, that the IDs of product attributes are alyways un-scoped, i.e. the ID for the insurance sum is InsuranceSum and not LifeProduct.InsuranceSum, as will be the ID of the resulting question for the insurance sum.

POST /rs/knowledgebases/example-kb/versions/1-0-0/productchange
{
    "products": {
        "LifeProduct": {
            "attributes": {
                "InsuranceSum": 200000
            }
        }
    },
    "riskAssessment": { ... }
}

The POST to the product change resource is answered with a HTTP status code of 200, having the changed risk assessment data in the response body, which in turn can then be used to further proceed with assessments.

{
    "language": "en",
    "completed": false,
    "valid": true,
    "products": [{"id": "LifeProduct", ... }],
    "questionnaire": {
        "questions": [
            {
                "id": "ApplicationDate",
                "scope": "GLOBAL",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": "2015-01-01",
                ...
            },
            {
                "id": "InsuredPerson.DateOfBirth",
                "scope": "ASSESSMENTFACTOR:InsuredPerson",
                "answerModification": { "source": "anonymous:Applicant", "timestamp": "2015-01-01T00:00:00" },
                "answer": "1975-10-02",
                ...
            },
            {
                "id": "LifeProduct.InsuranceSum",
                "scope": "INSURANCEPRODUCT:LifeProduct",
                "answerModification": { "source": "ProductChange:RAS-REST", "timestamp": "2016-01-01T00:00:00" },
                "answer": 200000,
                ...
            }
        ]
    },
    ...
}

As can be seen the “Life” product is now selected for risk assessment (see products property). Further, the questions ApplicationDate and InsuredPerson.DateOfBirth have been retained - together with their original answers and answer-modification information. A new question LifeProduct.InsuranceSum has been added, answered with the insurance sum amount of 200 000 which has been specified in the product configuration’s attributes. It’s answer-modification information has been set with the source that did answer the question ("ProductChange:RAS-REST") while the timestamp has been set to the time at which the product-change has been performed. The previously existing question CICProduct.InsuranceSum has been removed, since it is no longer relevant.

Scenario: Add rider

Given assessed risk assessment data for a “Life” product without any riders selected as shown below.

{
    "language": "en",
    "completed": false,
    "valid": true,
    "products": [{
        "id": "LifeProduct", 
        "riders": [], 
        ... 
    }],
    "questionnaire": {
        "questions": [
            {
                "id": "ApplicationDate",
                "scope": "GLOBAL",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": "2015-01-01",
                ...
            },
            {
                "id": "InsuredPerson.DateOfBirth",
                "scope": "ASSESSMENTFACTOR:InsuredPerson",
                "answerModification": { "source": "anonymous:Applicant", "timestamp": "2015-01-01T00:00:00" },
                "answer": "1975-10-02",
                ...
            },
            {
                "id": "LifeProduct.InsuranceSum",
                "scope": "INSURANCEPRODUCT:LifeProduct",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": 200000,
                ...
            }
        ]
    },
    ...
}

In order to add the “ADB” rider to the “Life” product a POST to the product change resource is performed as shown below.

Thereby, the riskAssessment property of the request body’s JSON document contains previously assessed risk assessment data shown above.

The products property contains the configuration for the “ADB” rider that is to be added. Thereby, an insurance sum of 50 000 is specified for the “ADB” rider by providing a corresponding ID/value pair in the optional attributes object. Note, that the IDs of rider attributes are alyways un-scoped, i.e. the ID for the insurance sum is InsuranceSum and not ADBRider.InsuranceSum, as will be the ID of the resulting question for the insurance sum. Note also, that no attributes for the “Life” product have been specified, because the corresponding question (i.e. LifeProduct.InsuranceSum) has already been answered in the given risk assessment data and therefore will be retained when changing the product configuration.

POST /rs/knowledgebases/example-kb/versions/1-0-0/productchange
{
    "products": {
        "LifeProduct": {
            "riders": {
                "ADBRider": {
                    "attributes": {
                        "InsuranceSum": 50000
                    }
                }
            } 
        }
    },
    "riskAssessment": { ... }
}

The POST to the product change resource is answered with a HTTP status code of 200, having the changed risk assessment data in the response body, which in turn can then be used to further proceed with assessments.

{
    "language": "en",
    "completed": false,
    "valid": true,
    "products": [{
        "id": "LifeProduct", 
        "riders": [{"id": "ADBRider", ... }], 
        ... 
    }],
    "questionnaire": {
        "questions": [
            {
                "id": "ApplicationDate",
                "scope": "GLOBAL",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": "2015-01-01",
                ...
            },
            {
                "id": "InsuredPerson.DateOfBirth",
                "scope": "ASSESSMENTFACTOR:InsuredPerson",
                "answerModification": { "source": "anonymous:Applicant", "timestamp": "2015-01-01T00:00:00" },
                "answer": "1975-10-02",
                ...
            },
            {
                "id": "LifeProduct.InsuranceSum",
                "scope": "INSURANCEPRODUCT:LifeProduct",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": 200000,
                ...
            },
            {
                "id": "ADBRider.InsuranceSum",
                "scope": "ADBRider",
                "answerModification": { "source": "ProductChange:RAS-REST", "timestamp": "2016-01-01T00:00:00" },
                "answer": 50000,
                ...
            }
        ]
    },
    ...
}

As can be seen the “ADB” rider is now selected for risk assessment (see property products → riders). Further, a new question ADBRider.InsuranceSum has been added, answered with the insurance sum amount of 50 000 which has been specified in the “ADB” rider’s configuration of attributes. It’s answer-modification information has been set with the source that did answer the question ("ProductChange:RAS-REST") while the timestamp has been set to the time at which the product-change has been performed. The question LifeProduct.InsuranceSum has been retained - together with its original answer and answer-modification information, because no InsuranceSum attribute has been specified in the “Life” product’s configuration. If the InsuranceSum attribute had been specified - even with the same answer as already present in the questionnaire - the answer-modification information for that question would have changed. In this case its source would have been set to "ProductChange:RAS-REST" and the timestamp would have been set to the time at which the product-change has been performed.

Scenario: Remove rider

Given assessed risk assessment data for a “Life” product with “ADB” rider as shown below.

{
    "language": "en",
    "completed": false,
    "valid": true,
    "products": [{
        "id": "LifeProduct", 
        "riders": [{"id": "ADBRider", ... }], 
        ... 
    }],
    "questionnaire": {
        "questions": [
            {
                "id": "ApplicationDate",
                "scope": "GLOBAL",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": "2015-01-01",
                ...
            },
            {
                "id": "InsuredPerson.DateOfBirth",
                "scope": "ASSESSMENTFACTOR:InsuredPerson",
                "answerModification": { "source": "anonymous:Applicant", "timestamp": "2015-01-01T00:00:00" },
                "answer": "1975-10-02",
                ...
            },
            {
                "id": "LifeProduct.InsuranceSum",
                "scope": "INSURANCEPRODUCT:LifeProduct",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": 200000,
                ...
            },
            {
                "id": "ADBRider.InsuranceSum",
                "scope": "ADBRider",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": 50000,
                ...
            }
        ]
    },
    ...
}

In order to remove the “ADB” rider from the “Life” product a POST to the product change resource is performed as shown below.

Thereby, the riskAssessment property of the request body’s JSON document contains previously assessed risk assessment data shown above.

The products property contains the configuration for the “Life” product with no riders specified (i.e. the riders property is an empty object, but we could as well have omitted the riders property completely). Note, that no attributes for the “Life” product have been specified, because the corresponding question (i.e. LifeProduct.InsuranceSum) has already been answered in the given risk assessment data and therefore will be retained when changing the product configuration.

{
    "products": {
        "LifeProduct": {
            "riders": {} 
        }
    },
    "riskAssessment": { ... }
}

The POST to the product change resource is answered with a HTTP status code of 200, having the changed risk assessment data in the response body, which in turn can then be used to further proceed with assessments.

{
    "language": "en",
    "completed": false,
    "valid": true,
    "products": [{
        "id": "LifeProduct", 
        ... 
    }],
    "questionnaire": {
        "questions": [
            {
                "id": "ApplicationDate",
                "scope": "GLOBAL",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": "2015-01-01",
                ...
            },
            {
                "id": "InsuredPerson.DateOfBirth",
                "scope": "ASSESSMENTFACTOR:InsuredPerson",
                "answerModification": { "source": "anonymous:Applicant", "timestamp": "2015-01-01T00:00:00" },
                "answer": "1975-10-02",
                ...
            },
            {
                "id": "LifeProduct.InsuranceSum",
                "scope": "INSURANCEPRODUCT:LifeProduct",
                "answerModification": { "source": "a:CustomerPortal", "timestamp": "2015-01-01T00:00:00" },
                "answer": 200000,
                ...
            }
        ]
    },
    ...
}

As can be seen the “ADB” rider is no longer selected for risk assessment (actually there is no riders property for the “Life” product any more). The previously existing question ADBRider.InsuranceSum has been removed, since it is no longer relevant. The question LifeProduct.InsuranceSum has been retained - together with its original answer and answer-modification information, because no InsuranceSum attribute has been specified in the “Life” product’s configuration. If the InsuranceSum attribute had been specified - even with the same answer as already present in the questionnaire - the answer-modification information for that question would have changed. In this case its source would have been set to "ProductChange:RAS-REST" and the timestamp woulf have been set to the time at which the product-change has been performed.

Important note: The removal of a rider only works if the rider is not marked as a mandatory rider of the product. By default, when changing a product configuration, the changed risk assessment data will have all mandatory riders selected for the specified products - even if those riders have not been explicitly specified in the new product configuration.
