Assessment factor search question with exposed unused factors

A question of type assessment factor search may expose unused factors as an attribute. This is indicated by the property exposesNextUnusedFactor, which is set to true. Unused factors can be used as answers to the corresponding search question, allowing to trigger the calculation of follow up questions without requiring the user to give real answers. It is therefore common to render this search question like a list question.

Suppose the user is faced with a question which can be answered multiple times. An example would be the measurements of the users blood pressure. Typically, a list question could be used to represent this kind of question, as every measurement consists of the same properties. There are however special cases in which each answer to such a question could consist if a different set of properties (see the beneficiary example below). A simple list question would not be sufficient, as it can not provide the different sets of reflexive questions required for the different possible answers.

Instead, every time the user wants to give an answer, he has to be asked about the type of answer he wants to give, followed by a set of reflexive questions. These questions can be seen as an embedded questionnaire, which as a whole is an answer to the original question. This can be achieved using a question of type assessment factor search, with its property exposesNextUnusedFactor set to true. The embedded questionnaire is modelled by a special set of assessment factors, provided via the nextUnusedFactor property. The question should therefore be rendered like a list question and allow the user to provide multiple, differently structured answers.

The following scenario describes the usage of the exposed factors to let the user enter multiple answers to a question, where every answer may consist of different reflexive questions.

Example: Add multiple beneficiaries to a life insurance product

Suppose the user wants to conclude a life insurance and is asked to give one or more beneficiaries, which may be a natural person or an organization or something completely different. The required information obviously differ from each other, depending on the type of beneficiary the user wants to provide. As explained above, a simple list question is insufficient in this case, since we need different information based on the type of beneficiary and therefore a different set of child questions. In order to provide this flexibility, a search question which exposes an unused factor is used. By answering the search question with this provided assessment factor the reflexive questions are triggered, in this case asking the user about the type of beneficiary he wants to add.

The following JSON snippet contains the unanswered assessment factor search question as it may appear within the questionnaire:

{
                "type": "assessment-factor-search",
                "id": "Product.BeneficiarySearch",
                "scope": "INSURANCEPRODUCT:Product",
                "caption": "Add Beneficiary",
                "tags": { ... },
                "constraints": {
                    "required": true
                },
                "meta": {
                    "ras": {
                        "questionMapping": {
                            "sourceAssessmentVariableType": "AssessmentFactorSearchVariable"
                        }
                    }
                },
                "index": "1",
                "isNew": true,
                "hasReflexive": true,
                "assignedTo": "Applicant",
                "initialAnswerByProductChange": false,
                "links": [
                    ...
                ],
                "searchId": "BeneficiarySearch",
                "searchLanguage": "en",
                "nextUnusedFactor": {
                    "id": "Beneficiary1"
                },
                "exposesNextUnusedFactor": true,
                "answerState": "missing"
            }

It is captioned with “Add Beneficiary”, because it serves as a starting point to enter a new beneficiary. In this case, as the property exposesNextUnusedFactor is set to true, the question is most likely not rendered like a normal search question but a list. If a nextUnusedFactor is provided, an “add” button should be rendered next to it. If no next unused factor is provided, it is not possible to add another answer to the question. By clicking on the “add” button, the user adds the nextUnusedFactor, in this case “Beneficiary1”, as an answer to the search question. A following round trip will then trigger the calculation of the reflexive questions, asking the user whether he wants to add a person or an organization:
