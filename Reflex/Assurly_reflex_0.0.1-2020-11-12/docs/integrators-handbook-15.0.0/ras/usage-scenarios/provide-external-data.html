Provide external data

Scenarios:

  • Pre-selecting products

  • Answer questions (assessment factor search)

Scenario: Pre-selecting products

Products can be selected even before the risk assessment has started. The following example shows a provided data block with a products object that includes the “LifeProduct”. This product will be automatically added by the RAS-REST-API when a risk assessment is performed.

POST /rs/knowledgebases/{kb}/versions/{version}/riskAssessments
{
  "language": "en",
  "provided": [
    {
      "source": {"id": "a", "type": "CustomerPortal"},
      "products": {
        "LifeProduct": {}
      }
    }
  ]
}

The API returns the updated risk assessment data with the added product. The mandatory riders are automatically added.

{
  "language": "en",
  "completed": false,
  "valid": true,
  "products": [
    {
      "id": "LifeProduct",
      "caption": "Life product",
      "riders": [
        {
          "id": "TPDownRider",
          "caption": "TDP-own rider",
          "mandatory": true
        },
        {
          "id": "CICRider",
          "caption": "CIC rider",
          "mandatory": true
        }
      ],
      "typeOfRiskCovered": "LifeTerm"
    }
  ],
  ...
  "provided": [
    {
      "source": {"id": "a", "type": "CustomerPortal"},
      "products": {
        "LifeProduct": {}
      }
    }
  ]
}

It is also possible for clients to specify which riders are used. The riders object can be used to define the riders:

POST /rs/knowledgebases/{kb}/versions/{version}/riskAssessments
{
  "language": "en",
  "provided": [
    {
      "source": {"id": "a", "type": "CustomerPortal"},
      "products": {
        "LifeProduct": {
          "riders": {
            "ADBRider": {}
          }
        }
      }
    }
  ]
}

The API returns the updated risk assessment data with the added product and riders. The mandatory riders are automatically added. Therefore, the risk assessment has three riders:

  • the mandatory riders TPDownRider and CICRider,
  • and the optional rider ADBRider.
{
  "language": "en",
  "completed": false,
  "valid": true,
  "products": [
    {
      "id": "LifeProduct",
      "caption": "Life product",
      "riders": [
        {
          "id": "TPDownRider",
          "caption": "TDP-own rider",
          "mandatory": true
        },
        {
          "id": "CICRider",
          "caption": "CIC rider",
          "mandatory": true
        },
        {
          "id": "ADBRider",
          "caption": "ADB rider",
          "mandatory": false
        }
      ],
      "typeOfRiskCovered": "LifeTerm"
    }
  ],
  ...
  "provided": [
    {
      "source": {"id": "a", "type": "CustomerPortal"},
      "products": {
        "LifeProduct": {
          "riders": {
            "ADBRider": {}
          }
        }
      }
    }
  ]
}

Scenario: Answer questions (general)

The provided data section can also be used to define answers for questions before the API has asked for this information. A client can therefore put together information even before starting the communication with the RAS-REST-API.

Clients needs to know the assessment variable ids (question ids) and the type of the questions (text, number, date, selection, etc.).

POST /rs/knowledgebases/{kb}/versions/{version}/riskAssessments
{
  "language": "en",
  "provided": [
    {
      "timestamp": "2015-01-01T13:59:59+01:00",
      "source": {"id": "a", "type": "CustomerPortal"},
      "questionnaire": {
        "NicotineDependence.NumberOfCigarettesPerDay": {
          "answer": 10
        },
        "IsSmoker": {
          "answer": "Yes"
        },
        "InsuredPerson.Height": {
          "answer": 185
        },
        "ApplicationDate": {
          "answer": "2015-01-01"
        },
        "NicotineDependence.TypesOfSmoke": {
          "answer": [
            "Cigarettes"
          ]
        }
      }
    }
  ]
}

In this example the client provides multiple answers. Reflexive questions will be answered within the same request. The first question “IsSmoker” will be answered automatically with the selection option “Yes”. This answer will lead to further questions related to NicotineDependence.

Note: It is also possible to provide a unit together with the value of a question. The provided unit must be available and supported by the question, otherwise the value is not applied and an error message is attached to it. If no unit is provided, but the question expects the answer to have one, the default unit of the question is automatically selected.

Scenario: Answer questions (assessment factor search)

An assessment factor search question can be answered from a provided data block either by providing IDs or labels of assessment factors. When providing labels of assessment factors, an additional answer descriptor is required. In the following you find examples of answering assessment factor search questions by providing IDs and labels for assessment factors.

Using assessment factor IDs

POST /rs/knowledgebases/{kb}/versions/{version}/riskAssessments
{
    "products": [...],
    "language": "en",
    "provided": [
        {
            "source": {"id": "a", "type": "CustomerPortal"},
            "questionnaire": {
                "Occupations": {
                    "answer": ["SkyDiver", "ScubaDiver"]
                } 
            }
        }
    ]
}

In the example above, IDs of assessment factors have been provided for answering the assessment factor search question “Occupations”. Thereby, no answer descriptor has been provided. This is the default behaviour: When no answer descriptor is provided, the supplied answer values are interpreted as being IDs of assessment factors.

In the following example an answer descriptor is used to explicitly specify that IDs of assessment factors have been provided. Note, that both examples are absolutely equivalent.

POST /rs/knowledgebases/{kb}/versions/{version}/riskAssessments
{
    "products": [...],
    "language": "en",
    "provided": [
        {
            "source": {"id": "a", "type": "CustomerPortal"},
            "questionnaire": {
                "Occupations": {
                    "answer": ["SkyDiver", "ScubaDiver"],
                    "answerDescriptor": "Id"
                } 
            }
        }
    ]
}

Using assessment factor labels

POST /rs/knowledgebases/{kb}/versions/{version}/riskAssessments
{
    "products": [...],
    "language": "en",
    "provided": [
        {
            "source": {"id": "a", "type": "CustomerPortal"},
            "questionnaire": {
                "Occupations": {
                    "answer": ["sky diver", "scuba diver"],
                    "answerDescriptor": "Label"
                } 
            }
        }
    ]
}

In the example above, labels of assessment factors have been provided for answering the assessment factor search question “Occupations”. Thereby, an answer descriptor has been provided, specifying that the provided answer values are labels of assessment factors.

Scenario: Adding assessment factors

Clients can add assessment factors to risk assessments without answering questions. A provided data object has an optional array assessmentFactors. The items of this array identify assessment factors with id and category. If the id is not found or the category does not exist, then the provided assessment factor is ignored and not added.

POST /rs/knowledgebases/{kb}/versions/{version}/riskAssessments
{
  "language": "en",
  "provided": [
    {
      "source": {"id": "a", "type": "CustomerPortal"},
      "assessmentFactors": [
        {
          "id": "Asthma",
          "category": "Medical"
        },
        {
          "id": "NicotineDependence",
          "category": "Medical"
        }
      ]
    }
  ]
}

Scenario: Removing answers

If clients remove answers in the questionnaire, then RAS-REST tries to automatically re-answer the questions with provided answers. If however the question in the questionnaire has an answerModification, then the question will not be answered automatically by RAS-REST - as long as the provided answer has no overwrite-mode or the overwrite-mode prohibits the modification of the question’s answer when provided answers are processed. This allows applicants to correct / change a question’s answer even though an answer for the question has been provided in a provided data block. Scenarios for explicitly update or overwrite answers from provided data blocks are described in “Update Externally Provided Data”.

"questions": [
  {
    "type": "date",
    "id": "ApplicationDate",
    "scope": "GLOBAL",
    "caption": "Application date",
    "precision": "day",
    "answerModification": {
       "source": "anonymous:Applicant",
       "timestamp": "2015-11-27T13:48:46+01:00"
     },
     ...
  }
]
