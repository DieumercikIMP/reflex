Architecture

Context model

RAS integration scenario

With the RAS integration scenario only the Risk Assessment Service (RAS), including the knowledge base, and the optional Document Creation Service (DCS), are deployed to the client’s site. All UI and end user interactions and access to third party services need to be implemented as part of the client integration project. The separate deployment of the RAS and the DCS is technically possible, as displayed in the context model diagram below. Nevertheless, it is suggested to deploy both components to the same application server. Besides the RAS and the DCS deployment artefacts that expose RESTful services, there need to be several configuration files and the knowledge base present in the filesystem. For further configuration details, refer to the subsequent chapters of this document. Components inside the RAS and the DCS containers refer to RESTful services. The Management Information Data Warehouse (MI DWH) is a service provided and managed by Hannover Re. AppQueue is the component to upload data to the MI DWH, it must be configured to run at the client’s site.

CEP integration scenario

In addition to the components that need to be deployed and configured as part of the RAS integration scenario, this scenario includes the CEP backend and the CEP front end. Besides the RESTful CEP backend services, the CEP backend can be configured to use data stores for the internal suspend and resume mechanism and to persist active cases. The CEP front end is a set of static files that need to be served by an application- or web-server other than the CEP backend application server. It is suggested to place those files on the same server that serves the client’s sales system portal front end.

RSP integration scenario

RSP is delivered with front end and back end services in addition to CEP. Please see CEP notes above.

Runtime model

RAS integration scenario

The sequence diagram describes the high-level happy path started by an end user, including the upload of the assessment results to the MI DWH. If third party data services/ external databases need to be accessed to include external data into the risk assessment, all external calls need to be implemented at the client’s sales system. External data can be included as so-called “provided data” as part of the initial request to the RAS component. Another use of provided data is to provide answers to questions that would have been asked as part of the risk assessment questionnaire.

CEP integration scenario

By including CEP into the system, all requests to the RAS component are filtered by the CEP backend. And before the risk assessment can be initiated as described in the RAS integration scenario, there needs to be a very first initialization step to initialize the CEP usage. This step includes the call to the integration service, which is part of the CEP backend, to create a JSON Web Token (JWT) that will be passed back and forth between the UI and the backend to identify and authorise the current user.

RSP integration scenario

The same principles mentioned in the previous section apply to this scenario.

Deployment model blueprints

RAS integration scenario

To ensure security and stability it is suggested to use common web hosting best-practices that implement a multi-layered encapsulation of all ReFlex components. No direct access to any ReFlex resource is allowed, all connections and communications must be filtered and secured by

  • a reverse proxy to restrict access on the application level
  • a firewall to control the network and transport layer

CEP integration scenario

The same principles mentioned in the previous section apply to this scenario. The difference is that there needs to be a second application server to host the Client’s Sales System together with the static files that represent the CEP frontend.

The AppQueue component should reside on the same server as the data store holding completed cases, or at least in a zone that is accessable to some kind of export mechanism, so that completed cases data records can be handed over.

RSP integration scenario

The same principles mentioned in the previous section apply to this scenario.
