Integration scenarios

ReFlex is a flexible and modularised software ecosystem. As such it provides a couple of setups in which different combinations of ReFlex components can be used in order to create a working system.

Basic integration scenarios

1. RAS integration scenario

This scenario is the minimal setup that is needed in order to build a ReFlex system. It contains all components that are essentially needed for a ReFlex system: RAS and a KB. The RAS acts as a runtime environment for the KB and is able to drive the questionnaire. The KB has to be simply deployed within the RAS to make the rule logic available via RAS REST API. The UI needed for end users has to be provided / created completely by the client or the client integration project (is not part of Hannover Re’s responsibility).

2. CEP integration scenario

In this scenario the ReFlex UI framework CEP is used within the client integration in addition to the RAS backend services.

This scenario builds on the RAS integration scenario where ReFlex services are available via the RAS REST API. In order to provide ReFlex UI oriented services, an additional layer comes into play: the CEP Services. They reside on top of the RAS REST API and provide all the services needed for the CEP frontend part. The CEP frontend is based on JavaScript/AngularJS and provides a questionnaire which needs to be integrated into the portal of the client.

3. RSP integration scenario

RSP combines the aforementioned integration scenarios to an out-of-the-box-stack. RSP is a component that is designed for clients whose current state of digitalization does not match the RAS or CEP integration scenarios. RSP provides a web portal that is embedded into an existing sales process. It handles all of the steps required for a risk assessment in order to make it easy for a cedent to quickly replace a paper-based health declaration. The focus of RSP is a quick and easy way to introduce automated underwriting without having to dedicate significant time and resources to integration. RSP contains the core ReFlex features that are ready to use with minimal integration and customization.

Only few customization steps have to be considered when RSP integration scenario is selected:

  • Adapting the “look and feel” to the cedent’s specifications
  • Adapting the terms and conditions to the cedent’s specifications
  • Integrating authentication and output interfaces
  • Creating documentation templates (PDF)
  • Integration into the sales portal
  • Integration into other cedent systems

Optional components

The three aforementioned integration scenarios can be enhanced by some more optional ReFlex components. The optional components are:

  1. DCS: This backend component is used for generating documents (e.g. PDF, XML) out of the assessment data structure (JSON). Apart from the integration with CEP, it can also be used directly by the client’s portal backend or frontend. If a client does not need document generation or wants to create documents using a different approach the component DCS can be skipped for an integration scenario.

  2. 3PS: This is a framework that integrates with external data providers to aid in insurance risk assessment. ReFlex communicates with Third Party Service providers through 3PS connectors which have to be implemented by the integrator. Each 3PS connector manages the request-response life-cycle of a single service provider. The response from a service provider is aligned with assessment factors and assessment variables by the 3PS adapter. The final response sent to the client includes assessment factors, assessment variables, and a base64 encoded string of the raw service provider response.

  3. AppQueue/MI: ReFlex includes a real-time management information component that provides the required level of detail for operative and strategic decisions. Ad-hoc reporting and analysis tools provide insights into distribution channel performance and periodic sales figures, the effectiveness of underwriting based on statistical evidence, recommended changes to a customer’s journey and probable causes for claims and any other risk indicators. The MI itself will not be part of a delivery since it is hosted by Hannover Re. AppQueue is a thin client that will anonymise and transfer completed applications from the client’s system to the MI at Hannover Re.

  4. UWB: ReFlex UWB is a component for the manual underwriting of incoming applications and assessment of incoming claims. It is a stand-alone, web-based application which can be tailored to the insurer‘s needs via many configuration options. ReFlex UWB also provides technical interfaces to import application and claims cases automatically (e.g. from the ReFlex RAS automated risk assessment component) and to retrieve the underwriting decision. Also, UWB can retrieve and process information from the ReFlex 3PS Connectors MIB (US and Canada only) and LNRC (US only).
