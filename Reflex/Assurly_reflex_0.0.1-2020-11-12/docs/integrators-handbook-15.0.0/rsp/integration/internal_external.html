Internal vs. External Prepare Integration Scenarios

RSP offers two integration scenarios for the creation of the user and the assessments.

Internal Prepare

User is created through the endpoint /rsp/user/create.

Assessments can be created (after proper authentication) using the initial information page that implements the product selection (it uses internally a call to /rsp/user/{userContextId}/internal-prepare).

Characteristics:

  • Limited functionality to provide external data for assessments in the RSP.
  • Product selection can be done inside RSP. The user will be able to manage the assessments.
  • Multiple products per user (multiple assessments per user).

External Prepare

User and assessment are created through the endpoint /rsp/user/prepare.

Characteristics:

  • Data for initialization has to be provided to the call.
  • The call must be performed from outside the RSP.
  • Product selection must be performed outside the RSP (thus, implemented). The user won’t be able to manage the assessments.
  • Multiple products per user (one assessment per each call to /rsp/user/prepare).

Provided Data in the RSP

Data can be provided to the RSP from the outside in two ways:

  • Internal prepare scenario: /rsp/user/create endpoint is called from the outside with a PrepareInformation object. The provided data contained herein is written into the created user. /rsp/assessment/{assessmentWrapperId}/initialize always uses the provided data block to initialize the assessment.
  • External prepare scenario: /rsp/user/prepare endpoint is called with a PrepareInformation object. The provided data contained herein is written into the created assessment. /rsp/assessment/{assessmentWrapperId}/initialize always uses the provided data block to initialize the assessment.

Note: In previous versions of the RSP the initialization endpoint was inconsistently defined as /rsp/assessment/{assessmentWrapperId}/initialise using British English.

PrepareInformation Object Example

Both /rsp/user/create as well as /rsp/user/prepare are called with a PrepareInformation body. TrackingId, partnerId and loggedInMessage are always stored in the user. Depending on the call, the provided data block is put into the created object. Since /rsp/user/create only creates a user, the provided data block is stored there. With /rsp/user/prepare, the provided data block is stored in the assessment.

{
  "trackingId": "3",
  "partnerId": "00000",
  "loggedInMessage": "Niko Laine was here",
  "providedData": {
   "source": {
    "id": "NordicDemo",
    "type": "CustomerPortal"
   },
   "products":{
    "CIProduct": {}
     
   },
   "questionnaire":{
             "InsuredPerson.FirstName": {
                 "answer": "Niko"
             },
             "InsuredPerson.LastName": {
                 "answer": "Laine"
             },
             "InsuredPerson.DateOfBirth": {
                 "answer": "1979-05-23"
             },
             "InsuredPerson.Gender": {
                 "answer": "Male"
             }
   }
  }
 }

Usage of PrepareInformation Object

  • /rsp/user/prepare: values for trackingId and partnerId are mandatory. All members of provided data (source, products, questionnaire) are required, but are allowed to have empty values.
  • /rsp/user/create: values for trackingId and partnerId are mandatory. Of the provided data member, only the questionnaire is ever used. It is copied into the assessment when /rsp/user/{userContextId}/internal-prepare creates it.

Multiple Calls to /rsp/user/prepare or /rsp/user/create

Both endpoints identify and return a user by trackingId and partnerId. Once the user is created, it is not overridden or changed. Therefore, calling /rsp/user/create twice with the same trackingId and partnerId has no effect. Calling /rsp/user/prepare twice with the same trackingId and partnerId will create a second assessment for the user using the provided data block of the call.
