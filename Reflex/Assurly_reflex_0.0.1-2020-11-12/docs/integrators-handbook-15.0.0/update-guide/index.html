Update guide

It is recommended to remove any previous ReFlex installation before installing a newer version. To visualise the removal of an existing ReFlex installation this guide shows the necessary steps for an Apache Tomcat server installation.

  • Stop the tomcat by executing <server.home.dir>/bin/shutdown.sh on a UNIX like OS or execute <server.home.dir>\bin\shutdown.bat on a Windows operating system.
  • Remove all ReFlex artefacts as well as their respective exploded war directories from <server.home.dir>/webapps.
  • Remove all ReFlex configuration directories from the configured REFLEX_CONFIG_ROOTDIR (e.g. <server.home.dir>/reflex-config).
  • Proceed with the installation guide that describes the ReFlex installation for a clean environment.

Configuration overwrite

To keep specific local configurations even when upgrading ReFlex to newer versions it is possible to overwrite single lines of the config.properties for RAS or CEP with help of environment property.

For example to overwrite the CEP config parameter

com.hannoverre.reflex.cep.util.ras.user=user

you can set the environment property

COM_HANNOVERRE_REFLEX_CEP_UTIL_RAS_USER=someotheruser

The environment property will overwrite the value of the original config file.
For more information about environment variables please have a look at the installation guide step 3 configurations.

Migration guide

The package that was delivered by Hannover Re contains all the artefacts required to update your existing ReFlex installation. Depending on the update scenario that addresses your specific ReFlex integration it might be required to perform some migration steps after updating the ReFlex components.

Although RAS may be a stateless system after its integration into the client’s systems, the complete environment may not be. As soon as one system of the environment stores non-assessed applications it is most likely that an update path has to be chosen to assure that the already created applications can still be assessed in the future.

The decision which path has to be pursued depends on the delivered artefact(s) as well as on the system that should be used to assess the applications. Depending on the decision whether the old components or the newly delivered ones should be used for that purpose, the integrator has to perform different migration steps.

This chapter lists different scenarios bases of the delivered artefact(s). For detailed update information from one ReFlex release to the next, please have a look at the release and migration notes.

  • A new RAS and a new knowledge base have been delivered. There are no special remarks concerning potential update and migration scenarios in case a new RAS and a new knowledge base have been delivered. In this case, it is sufficient to follow the notes from the RAS chapter. After this, the corresponding usage scenario inside the knowledge base chapter should be read.
  • A new CEP and a new RAS have been delivered. If the update delivery contains a new RAS and CEP, the update paths of these components can be chosen. The instructions in the RAS chapter should be followed first. The CEP instructions should be read last.
  • A complete ReFlex software stack has been delivered. In case that the complete ReFlex software stack is updated the upgrade instructions of all components have to be followed in this specific order:
    • RAS update scenario
    • Knowledge base update scenario
    • CEP update scenario
