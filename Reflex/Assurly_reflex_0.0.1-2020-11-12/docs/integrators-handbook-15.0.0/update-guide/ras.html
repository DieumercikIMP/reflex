RAS update

This page lists the possible usage scenarios that might be considered if a new RAS middleware has been delivered. These scenarios are based on the kind of applications that might be handled inside the integrating system. That implies that the already existing knowledge base will be reused.

# kind of application that should be assessed RAS used for assessment RAS patch release RAS minor release RAS major release
1 newly created newly delivered OK recompile existing KB recompile existing KB
2 already existing newly delivered OK recompile existing KB recompile existing KB and follow technical migration steps if available
3 already existing old RAS that was used to create the case this use case is not supported; if this is required the whole ReFlex software stack has to be duplicated; inside the RAS integration scenario the client system has to make sure, that existing cases are assessed by the legacy system see patch release see patch release

The complexity of these scenarios is increased by the kind of RAS update that has been delivered. The first line describes the points that have to be regarded if cases will be created and assessed by the means of this new RAS middleware. If the update was a patch release there are no further activities required. Simply follow the update instructions of the release. For all the other releases (minor or major) it is required to use recompiled versions of your existing knowledge bases. It is not possible to use the existing knowledge bases with the new middleware.

The second scenario in which cases that have been created with an older version of the middleware should be assessed by using the new middleware is similar to the previous one. Only if the delivery contains a major update, it might be required to perform some migration steps. Depending on the particular changes within the release it might be necessary to patch the application JSON files before sending them to the new RAS middleware. A detailed description of those steps will be listed in the migration notes in the appendix of this document.

Should there be a requirement to assess existing cases with the former version of the RAS middleware as it is described in scenario 3 it has to be stated that there are no supported migration paths available. In case it would be required anyhow - maybe because due to legal issues - it is proposed to duplicate the whole ReFlex software stack. In the RAS integration scenario, this approach would be inherent to the requirement because the RAS would be the only ReFlex software component required for the risk assessment anyways. Should the client use the CEP integration scenario it is required to duplicate the used version of CEP as well because CEP only supports the usage of one RAS at a time.
