Up-, cross- and down-selling

Overview

In a standard sales process, the end user selects a product and optionally adds one or multiple riders that are available for the selected product. The end user then proceeds through the sales process and answers several questions that are asked by ReFlex based on the selected product and rider combination. At the end of the process, ReFlex provides the assessment result that depends on the selected product and answers given by the end user. At this point, an up-, cross- or down- selling option might be proposed to the end user that is based on the selected product/riders and decision.

Technically up-, cross- and down- selling are all different kinds of product/rider selection changes. To optimise this process ReFlex allows reusing already given answers by the end user so that same questions do not have to be answered again.

Up-selling

Up-selling means that the end user changes their selection to a more valuable selection by exchanging the selected product with a more valuable one, exchanging the rider with a more valuable one or adding a not yet selected rider that is available for the current product selection.

Down-selling

Down-selling is the opposite of up-selling and means that the end user exchanges the product/rider to a less valuable one or removes a rider. This might make sense if the end user was declined but would be accepted for a less valuable product/rider or only the product when removing the rider option.

Cross-selling

When the end user has finished the purchase of the initially selected product one might offer them the option to purchase a totally different product that covers some risk that was not covered by the previously selected product. In this case, a new application is initiated that contains the suggested product but it is still possible to carry over already given answers.

Technical solutions

Depending on the selected integration scenario (CEP or RAS), there are different solutions to implement up-, cross- and down-selling. In both integration scenarios, the new product/rider selection configuration needs to be provided via a REST API call.

Restrictions

ReFlex itself does not present up-, cross- or downselling options. The rules which new products/riders can be offered to the end customer needs to be determined by the integrating application.

Provided data

When using provided data, the integrating application needs to take care of transferring the data to the newly selected product. The insurance sum, for instance, is usually data that is bound to the product scope. If the product is changed, the scope changes as well and the integrating application needs to take care to provide a new or transfer the previously used insurance sum. The actual solution depends on the business requirements. Therefore all product switching APIs offer the possibility to update the provided data block. See Provide External Data for details.

RAS integration scenario

When integrating only RAS, the generic Product-Change REST resource can be used for all scenarios. See Change Product Configuration for details.

CEP integration scenario

When integrating the full ReFlex stack including the CEP, the product switch will be achieved by using two different REST resources depending on the scenario. Those two resources add some convenience to the low-level RAS API, which needs to be implemented by the integrating application when using the RAS integration scenario.

Up- and down-selling with CEP

Since up- or down-selling will modify the currently active application it can be done using the CEP’s integration-service Update application REST resource.

Cross-selling with CEP

When performing a cross-selling, the previous application will be finished by the end user and a new application needs to be created. This functionality is provided by the CEP’s integration-service Copy application REST resource.
