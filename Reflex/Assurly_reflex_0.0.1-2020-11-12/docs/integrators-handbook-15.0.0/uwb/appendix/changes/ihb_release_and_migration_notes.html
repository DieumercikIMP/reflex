Dashboard view count

  • Type: change
  • Migration: no manual migration steps required

UWB now displays the number of cases for each dashboard quick view in parenthesis behind the quick view’s name.

For example: “To Do (12)     Pending (14)     All Cases (258)”

The count shows the number of cases which would be shown if the associated quick view is selected. If search criteria have been entered (e.g. search by insured person’s name), the quick view count shows the resulting number of cases considering the search criteria.

Policy-in-force information on case level

  • Type: change
  • Migration: no manual migration steps required

The policy in force information (yes/no and optional date when policy went into force) can now be specified on the case level. Previously, it was specified on the insured person level.

For single life cases, this change does not have any effect.

However, for joint life cases this means that it is not possible anymore to specify that a policy has been issued to one person only but not to the other (which should never happen for ‘real’ joint life applications).

Unknown information about foreign stays

  • Type: change
  • Migration: no manual migration steps required

In the ‘geographical’ section of the risk details page, it is possible to specify for each foreign stay

  • whether the stay is longer than 6 months
  • and whether the stay is caused by a business trip or not.

Previously, it was only possible to answer “Yes” or “No” to the above questions.

Now, it is possible to explicitly specify that the information is unknown.

Default details on MIB form (US/Canada)

  • Type: change
  • Migration: no manual migration steps required

Only for US/Canada

The third-party data can now be retrieved from the MIB service using default settings for

  • MIB Holding Form (default: Individual)
  • and MIB Product Type (default: Term Life)

It is still possible to change these settings before retrieving the third-party data for an individual case.

User-specific dashboard views

  • Type: change
  • Migration: no manual migration steps required

User-specific dashboard views have been introduced. The DASHBOARD_VIEWS database table has been renamed to OFFICE_DASHBOARD_VIEWS. Additionally, a new database table USER_DASHBOARD_VIEWS has been added.

See the Dashboard Views page for more details.

New security configuration

  • Type: change
  • Migration: manual migration required (see below)

The configuration for the Authorization Server (/Keycloak) has changed.

Migration notes: The following parameters have to be added or changed in the configuration file:

  • The property com.hannoverre.reflex.umac.security.oidc.configurationUrl has been removed.
  • The property com.hannoverre.workbench.security.provider.issuer has been added and is required. It has to name the issuer URI of the authorization server. This is likely a prefix of the old com.hannoverre.reflex.umac.security.oidc.configurationUrl property.
  • The property com.hannoverre.workbench.security.provider.jwksUri has been added and is required. It has to contain the json keystore URL of the authorization server.
  • The property com.hannoverre.workbench.security.provider.authorizationEndpoint has been added and is required. It has to contain the authorization endpoint URL of the authorization server.
  • The property com.hannoverre.workbench.security.provider.tokenEndpoint has been added and is required. It has to contain the token endpoint URL of the authorization server.
  • The property com.hannoverre.workbench.security.provider.userInfoEndpoint has been added and is required. It has to contain the userinfo endpoint URL of the authorization server.
  • The property com.hannoverre.workbench.security.provider.endSessionEndpoint has been added and is required. It has to contain the endsession endpoint URL of the authorization server.
  • The property com.hannoverre.reflex.umac.security.allowedClockSkew has been removed.
  • The property com.hannoverre.workbench.ui.oauth.host has been removed. It is now derived from the resourceserver.jwt.issuer-uri property.
  • The property spring.security.oauth2.resourceserver.jwt.jwk-set-uri has been added and is optional. If known in advance, it can be added to make the UWB Backend able to start while the authorization server is unavailable, which increases overall system resilience.
  • The property com.hannoverre.reflex.umac.security.clientId has been renamed to com.hannoverre.workbench.security.clientId.
  • The property com.hannoverre.reflex.umac.security.uiClientId has been renamed to com.hannoverre.workbench.security.uiClientId.
  • The property com.hannoverre.reflex.umac.security.jwsAlgorithmWhiteList has been removed. The algorithms now default to Spring Boot and Spring Security’s defaults and can be configured using the according mechanism provided by those frameworks.

See the Authorization server installation page and the Application configuration page for more details.

New admin permissions

  • Type: change
  • Migration: manual migration required (see below)

The previous ADMIN permission which covered all administrative features in the admin portal has been split up into several, more fine-granular permissions:

  • ADMIN_MANAGE_CLIENT - manage the office configuration (excluding vendors), i.e. text modules, laboratory values, requirements…
  • ADMIN_MANAGE_VENDORS - manage vendors including their contacts and underwriting guidelines.
  • ADMIN_AUDIT_SAMPLE_REPORT - generate monthly audit report.
  • ADMIN_MANAGE_AUTHORITIES - manage underwriting authorities for office and users. (Note: this permission has been added for future improvements. Currently, the management of underwriting authorities is not yet possible via the admin portal.)

Migration notes: All existing user accounts / roles with the ‘ADMIN’ permission need to be adjusted to include the according more fina-granular permissions as listed above.

For example, an admin user who needs to be able to manage the list of vendors previously needed the permission ADMIN but now needs the permissions ADMIN and ADMIN_MANAGE_VENDORS.

Claims Module - introduced policy number

  • Type: change
  • Migration: no manual migration steps required

Introduced policy number for applications and claims.

For applications, the policy number can be entered optionally.

For claims, the policy number can be entered optionally or is mandatory, depending on the associated configuration settings.

Redesigned action bar

  • Type: change
  • Migration: no manual migration steps required

The actions buttons (e.g. for decision, decision letter, signing, etc.) in the action bar have been re-designed:

  • Related actions are now grouped together. For example, all signing actions (“Sign & Close”, “Sign & Request 2nd Signature”, “Request for Signature” and “Refer for Signing”) can now be found in action group “Signing”.
  • If an action group only contains a single action, that single action is shown instead of the group.
  • The action “Refer for Signing” has been renamed to “Refer to target office for Signing”.
  • The action “Refer for Opinion” has been renamed to “Refer to target office for Opinion”.

User profile / user settings

  • Type: change
  • Migration: no manual migration steps required

Every user can now manage her/his own settings without affecting other users. Initially, there are two user-specific settings:

  • Go directly to the Dashboard after closing/signing/referring/mark - if enabled, the user is taken back to the Dashboard automatically after closing a case, signing a case, requesting a signature for a case or marking a case as pending.
  • Open assessment factor dialog automatically - if enabled, the rating dialog is opened automatically when an impairment, avocational activity or residential risk is added.

Delete cases

  • Type: change
  • Migration: no manual migration steps required

It is now possible to delete cases physically from the database. Cases to delete are identified by their internal reference number (UWB number).

A deletion record is written with some technical information about the deleted case. The deletion record does not contain any personal information about the application/claim. Currently, the deletion record is not shown to the user and cannot be exported. This functionality will be implemented in one of the next releases.

Technical note: Deleting a case is an admin functionality and requires the special permission ADMIN_DELETE_CASES. See the Permission configuration page for more details.

Assign case to user

  • Type: change
  • Migration: no manual migration steps required

A user with the according permission may now assign one or several cases to another underwriter. There are two options:

  • select cases on the dashboard and assign them to an underwriter
  • open a case and assign it to an underwriter

Close without reason

  • Type: change
  • Migration: no manual migration steps required

If the signing workflow is disabled in the configuration, cases can now be closed without a reason.

If the signing workflow is enabled in the configuration, the behaviour is still the same: the user needs to enter aa reason why a case is closed without signing.

Excel report for claims and applications

  • Type: change
  • Migration: no manual migration steps required

It is now possible to export both claims and application cases into a single Excel sheet.

Use Authorization Code with PKCE

  • Type: change
  • Migration: manual migration required (see below)

The UWB user interface now uses the OAuth 2 Authorization Code Grant with PKCE. This may require adjustments in your authorization server configuration.

Migration Notes: Adjust your authorization server configuration to enable PKCE, see page Authorization Server Installation for further details.

Ratings in key facts

  • Type: change
  • Migration: no manual migration steps required

It is possible to configure the key facts to show ratings for occupation, avocational activities and residential/geographical risks. Additionally, it is possible to show build ratings (for BMI) and blood pressure ratings in the key facts as well. See section Key fact fields on the Key Facts page for further details.

Default lab values

  • Type: change
  • Migration: no manual migration steps required

All lab value definitions marked as default will be automatically added to the list in the lab value dialog. Only lab value for which a result has been captured are shown in the lab value table on the risk details page.

ReFlex import mapping - product attributes

  • Type: change
  • Migration: no manual migration steps required

The ReFlex JSON import mapping functionality has been extended in order to better support mapping products and their attributes (e.g. insurance sum, policy duration, etc.). Also, a new test selector has been introduced to improve filtering JSON subtrees for the mapping. See the ReFlex Import page, especially the shown mapping example.

ReFlex import - translations for risk assessment result

  • Type: change
  • Migration: no manual migration steps required

The translations for the risk assessment result delivered in the ReFlex JSON documents are now shown on the ReFlex page in ReFlex UWB for imported cases. This includes translations for

  • the decision
  • the decision attributes

UWB import API - examinations

  • Type: change
  • Migration: no manual migration steps required

The UWB import API has been extended to include examinations for the medical sections of the risk details. See the UWB Import API page for more details.

Dashboard quick views - renamed ‘cedent’ to ‘vendor’

  • Type: change
  • Migration: no manual migration steps required

The column ‘cedent’ has been renamed to ‘vendor’ in the dashboard quick view definition.

Migration notes: Existing dashoard quick views with this column are migrated automatically therefore no manual migration is needed.

MIB connector integration - zip code is now optional

  • Type: change
  • Migration: no manual migration steps required

The zip code is now an optional parameter for requesting third-party data from the MIB Group (US and Canada only).

Tags are now displayed in alphabetical order

  • Type: bug fix

TAGs assigned to a case are now shown in alphabetical order. Previously, the order was not deterministic.

Show full text in key facts section

  • Type: bug fix

Large text was not displayed correctly in the tooltips of the key facts section on the assessment summary page (the text was cut off when too long, but this was not visualised to the user).

Example: tooltip in key facts showing the impairment details.

This has been fixed. Now, the user will be able to see the full text entered.

Lower limit for examination’s date of completion

  • Type: bug fix
  • Migration: no manual migration steps required

Until now, it was possible to capture any date of completion in the past for examinations. The validation has been adjusted to accept only dates of completion after 01/01/1900. As before, a date in the future is not accepted.

Decision Letter dialog - show decision remarks

  • Type: bug fix
  • Migration: no manual migration steps required

Decision remarks were not shown in the decision letter dialog (however, printed on decision letter).

Blood pressure values validations systolic and diastolic

  • Type: bug fix
  • Migration: no manual migration steps required

When capturing blood pressure values in the medical section, it is now necessary to capture both systolic and diastolic values and the systolic value must be higher than the diastolic value.

Dashboard search with pagination

  • Type: bug fix
  • Migration: no manual migration steps required

Fixed two issues with the search / quick views on the Dashboard in combination with the pagination functionality:

  • search functionality did not work when not on first page of quick view
  • switching quick views did not work correctly when not on first page of quick view

Handling underwriting authorities for annuities

  • Type: bug fix
  • Migration: no manual migration steps required

Fixed the underwriting authority check for covers with regular (e.g. yearly) payments in combination with a flat underwriting authority limit.

Also eliminated currency conversion for the underwriting authority check if case and authority limits use the same currency (to avoid rounding errors).

User lock-out upon session expiration

  • Type: bug fix
  • Migration: no manual migration steps required

An issue has been fixed which caused the user to be locked out from UWB when the access token timed out.
