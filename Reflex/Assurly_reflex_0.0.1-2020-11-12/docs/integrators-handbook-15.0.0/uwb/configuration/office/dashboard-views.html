Dashboard views

Dashboard views are pre-configured filters and column-sets for the case list stored in the UWB database. They enable the user to easily filter the case list to show only relevant cases for certain use cases.

Common dashboard views are:

  • To do - show all unresolved cases requiring an action by the Underwriter, e.g. assessment, signing, finalisation, etc.
  • My cases - show all unresolved cases assigned to current user
  • Pending - show all cases for which additional information/documents have been requested from the Applicant.
  • All cases - show all stored cases

Dashboard views can be configured per office or per user. Office dashboard views are available to all users by default. Non-default dashboard views can be set up and assigned explicitly to selected users. If a user does not have specifically assigned dashboard views, the default office dashboard views are shown. If a user has specifically assigned dashboard views, only those are shown and none of the default office dashboard views.

To configure a dashboard view, the following attributes need to be set:

  • Label - the name as shown on the UWB dashboard
  • Position - the (numeric) position of the dashboard view button
  • Search - the search query including the filter and sort criteria and the case attributes / columns to be shown

Office dashboard views

The following SQL script can be used to set up office dashboard views:

insert into OFFICE_DASHBOARD_VIEWS (OFFICE_ID, POSITION, LABEL, SEARCH) values
   (<office-id>, 0, 'My Cases', '<search query>'),
   (<office-id>, 1, 'Pending', '<search query>'),
   (<office-id>, 2, 'All Cases', '<search query>');

The <office-id> is the database Id of the office record (office id). The <search query> contains an Elasticsearch type of query in JSON format. It consists of

  • a zero-based offset (“from”)
  • the filter query (“query”)
  • an optional sort criteria (“sort”)
  • the column list (“source”)

User-specific dashboard views

The following SQL script can be used to set up user-specific dashboard views:

insert into USER_DASHBOARD_VIEWS (USER_ID, OFFICE_ID, POSITION, LABEL, SEARCH) values
   (<user-id>, <office-id>, 0, 'Specific User View', '<search query>'),
   (<user-id>, <office-id>, 1, 'Another Specific User View', '<search query>');

The <user-id> is the database Id of the IWB_USER record.

The provided user-office combination must be valid, which means the given user must be assigned to the given office.

Please note that user-specific dashboard views can only be set up after the user has logged-in at least once. If a new user never has logged in, the user dashboard views cannot be linked by foreign key because the respective database entries are not available yet.

Search Query

Example configuration for the “My cases” dashboard view:

{
  "from": 0,
  "query": {
    "bool": {
      "must": [
        {
          "term": {
            "ownerId": "{user}"
          }
        }
      ],
      "should": [
        {
          "term": {
            "caseState": "IN_PROGRESS"
          }
        },
        {
          "term": {
            "caseState": "SIGNING"
          }
        },
        {
          "term": {
            "caseState": "SIGNED"
          }
        },
        {
          "term": {
            "caseState": "IN_REVIEW"
          }
        },
        {
          "term": {
            "caseState": "REVIEWED"
          }
        },
        {
          "range": {
            "followUpDate": {
              "lte": "{today}"
            }
          }
        }
      ],
      "minimum_should_match": 1
    }
  },
  "source": [
    "iwbNumber",
    "vendor",
    "externalReferenceNumber",
    "caseType",
    "insuredPersonDetails",
    "coverDetails",
    "caseState",
    "ownerName",
    "created"
  ]
}

“query” clause

The query clause supports the following operators and clauses:

Operator / Clause Description Example
term Matches a case attribute (string representation) Match only cases in state REOPENED:
{ “term”: { “caseState”: “REOPENED” } }
Match only cases assigned to the current underwriter*:
{ “term”: { “ownerId”: “{user}” } }
range Compares an attribute to a value (e.g. a fixed number or date), a case attribute (e.g. iwbNumber) or a supported expression.
Currently supported expressions for are:
- now (matching the current date and time)
- today (matching today’s date)
- yesterday (matching yesterday’s date)
- tomorrow (matching tomorrow’s date)
- endOfMonth (matching the current month’s last day)
Currently supported comparison operators are:
- gt (greater than)
- gte (greater than or equal)
- lt (less than)
- lte (less than or equal) 		Match only cases where the follow-up date has been reached:
{ “range”: { “followUpDate”: { “lte”: “{today}” } } }
exists Matches if the given attribute is set (not empty/null) Match only cases with an assigned underwriter: {“exists”:{“field”:“ownerName”}}
bool Subquery which may include ‘must’, ‘filter’, ‘should’, ‘must_not’ and ‘minimum_should_match’ sections.
Sub queries can be nested. 		See full example above.
must Combine criteria which must be fulfilled to deliver as a result. Only returns cases which are assigned to the current user:
“must”:[{“expr”:{“ownerId”:“currentUser”}
filter experimental - not fully supported yet not available
should Combine criteria which should be fulfilled to deliver as a result. Only returns cases which are either in state PENDING or CLOSED:
“should”:[{“term”:{“caseState”:“PENDING”}},{“term”:{“caseState”:“CLOSED”}}]
must_not Combine criteria which must not be fulfilled to deliver as a result. Only returns cases which have not yet been assigned to an Underwriter:
“must_not”:[{“exists”:{“field”:“ownerName”}}]
minimum_should_match Specifies the number of criteria in a should-clause which need to be fulfilled to deliver as a result.
If only a should-clause but no must-clause is configured, the query will only return cases which match at least one of the criteria in the should-clause. However, if combining a should-clause with a must-clause, the query will return cases which match none of the criteria in the should-clause. In order to return only cases matching the criteria in the must-clause and at least one* of the criteria in the should-clause, the minimum_should_match-clause must be added as well. 		At least one of the criteria in the associated should-clause must match: “minimum_should_match”: 1

“sort” clause

The sort element can be used to specify the sort criteria.

Example:

"sort":[{"lastUpdate":"DESC"}]

“source” clause

Supported columns (source) that can be displayed in the dashboard:

Note: Source colums are not the same as term or range query columns! They may partially overlap but are not always the same.

Attribute Description Required
iwbNumber UWB Number, e.g. HRE-TE-000001 with link to open case
caseType Type of case, e.g. “Application” (A) and “PreUnderwriting” (P) for Applications and “Claim” (C) and “Claim Opinion” (CO) for Claims
classification Case Classification, e.g. “SIMPLE”
accelerated Whether the case is an accelerated case or not
vendor Vendor/channel information (if used)
externalReferenceNumber External reference number
insuredPersonDetails Combined field including the full name of Insured Person followed by insured Person’s gender and insured Person’s date of birth
coverDetails Combined field including the name of first cover’s type of insurance followed by first cover’s insurance sum / face amount and cover duration (when defined)
caseState Current case state
policyInForce Case’s policy-in-force status
ownerName Assigned underwriter’s name
created Case creation date and time unless selected as sort criteria
lastUpdate Case’s most recent update / change date and time unless selected as sort criteria
signerName The first signing underwriter’s name
reviewerName The name of the underwriter who performed the review
followUpDate The follow-up date of the case (if set) with the follow-up text (if set)
policyNumber The policy number of the application or claim

Please note that the following details are aggregated into a single column and not each shown in a separate column:

  • Column “Insured Person” - Shows the first name, middle name and surname. Additionally, shows the gender and birthday.
  • Column “Cover Details” - Shows the name of the first cover’s type of insurance (Life, PHI, etc.) followed by its insurance sum including its currency and the benefit type ending with by the cover duration.
